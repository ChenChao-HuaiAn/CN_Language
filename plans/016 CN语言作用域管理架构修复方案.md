# 016 CN语言作用域管理架构修复方案

## 问题诊断

### 1. 当前架构分析

通过分析源代码，发现三个模块各自创建独立的作用域实例：

#### 1.1 scope_builder.c 的作用域管理

```c
// src/semantics/resolution/scope_builder.c:1601
function_scope = cn_sem_scope_new(CN_SEM_SCOPE_FUNCTION, parent_scope);
// ... 插入参数和局部变量符号 ...
cn_sem_build_block_scope(function_scope, function_decl->body, diagnostics);
// 函数返回后，function_scope 被丢弃
```

#### 1.2 semantic_passes.c 的作用域管理

```c
// src/semantics/checker/semantic_passes.c:34
CnSemScope *fn_scope = cn_sem_scope_new(CN_SEM_SCOPE_FUNCTION, global_scope);
// ... 重新插入参数和局部变量符号 ...
cn_sem_scope_free(fn_scope);  // 第46行：作用域被释放
```

#### 1.3 irgen.c 的作用域管理

```c
// src/ir/gen/irgen.c:2126
CnSemScope *func_scope = cn_sem_scope_new(CN_SEM_SCOPE_FUNCTION, parent_scope);
ctx->current_scope = func_scope;
cn_ir_gen_block(ctx, func->body);
cn_sem_scope_free(func_scope);  // 第2133行：作用域被释放
```

### 2. 问题根源

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        编译流程中的作用域生命周期                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  scope_builder.cn_sem_build_scopes()                                    │
│  ┌──────────────────────────────────────────┐                           │
│  │ global_scope (持久)                       │                           │
│  │   └── function_scope (临时，函数返回后丢弃) │ ◄── 问题1：符号丢失       │
│  │         └── block_scope (临时)            │                           │
│  └──────────────────────────────────────────┘                           │
│                      ↓                                                  │
│  semantic_passes.cn_sem_resolve_names()                                 │
│  ┌──────────────────────────────────────────┐                           │

│  │   └── fn_scope (新创建，无法访问上面的符号) │ ◄── 问题2：类型信息丢失   │
│  │         └── block_scope (新创建)          │                           │
│  └──────────────────────────────────────────┘                           │
│                      ↓                                                  │
│  irgen.cn_ir_gen_function()                                             │
│  ┌──────────────────────────────────────────┐                           │
│  │ global_scope (同一个实例)                  │                           │
│  │   └── func_scope (新创建，无法访问类型)    │ ◄── 问题3：生成错误C代码   │
│  └──────────────────────────────────────────┘                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3. 具体影响

1. **变量符号类型丢失**：`semantic_passes` 无法找到 `scope_builder` 插入的变量符号
2. **类型推导失败**：`infer_expr_type` 无法获取变量的类型信息
3. **C代码生成错误**：`irgen` 生成的C代码类型不正确

---

## 解决方案对比

### 方案 A：将作用域保存到 AST 节点

**实现方式**：在 `CnAstFunctionDecl` 和 `CnAstBlockStmt` 中添加 `owning_scope` 字段

**优点**：
- 作用域信息与AST节点绑定，生命周期清晰
- 符合"数据与节点共存"的设计原则
- 便于后续扩展（如调试信息、IDE支持）

**缺点**：
- 需要修改AST节点结构
- 需要修改AST释放逻辑

**代码修改量**：中等（约200行）

---

### 方案 B：复用 scope_builder 创建的作用域

**实现方式**：让 `semantic_passes` 和 `irgen` 使用 `scope_builder` 创建的作用域

**优点**：
- 不需要修改AST结构
- 代码修改量较小

**缺点**：
- 需要改变调用顺序
- 作用域生命周期管理复杂
- 可能影响其他模块

**代码修改量**：小（约100行）

---

### 方案 C：将类型信息保存到 AST 节点

**实现方式**：在 `CnAstVarDecl` 和 `CnAstExpr` 中添加 `resolved_type` 字段

**优点**：
- 类型信息直接可用，不需要查找作用域
- 便于后续优化

**缺点**：
- 需要修改AST节点结构
- 增加内存占用
- 不解决作用域管理的根本问题

**代码修改量**：大（约300行）

---

## 推荐方案：方案 A 的优化版本

### 设计原则

1. **最小侵入性**：只修改必要的AST节点
2. **生命周期清晰**：作用域随AST节点创建和释放
3. **向后兼容**：不破坏现有API

### 实现细节

#### 1. 修改 AST 节点定义

```c
// include/cnlang/frontend/ast.h

// 函数声明
typedef struct CnAstFunctionDecl {
    const char *name;
    size_t name_length;
    CnAstParameter *parameters;
    size_t parameter_count;
    struct CnType *return_type;
    CnAstBlockStmt *body;
    CnVisibility visibility;
    int is_interrupt_handler;
    uint32_t interrupt_vector;
    int is_prototype;
    int is_override;
    int is_static;
    
    // 【新增】函数作用域（由 scope_builder 创建，包含参数和局部变量符号）
    struct CnSemScope *owning_scope;
} CnAstFunctionDecl;

// 语句块
typedef struct CnAstBlockStmt {
    size_t stmt_count;
    struct CnAstStmt **stmts;
    
    // 【新增】块作用域（可选，用于嵌套块）
    struct CnSemScope *owning_scope;
} CnAstBlockStmt;
```

#### 2. 修改 scope_builder.c

```c
// src/semantics/resolution/scope_builder.c

static void cn_sem_build_function_scope(CnSemScope *parent_scope,
                                        CnAstFunctionDecl *function_decl,
                                        CnDiagnostics *diagnostics)
{
    CnSemScope *function_scope;
    
    function_scope = cn_sem_scope_new(CN_SEM_SCOPE_FUNCTION, parent_scope);
    if (!function_scope) {
        return;
    }
    
    cn_sem_scope_set_name(function_scope, function_decl->name, function_decl->name_length);
    
    // 【关键修改】将作用域保存到 AST 节点
    function_decl->owning_scope = function_scope;
    
    // 插入参数符号...
    for (size_t i = 0; i < function_decl->parameter_count; ++i) {
        CnAstParameter *param = &function_decl->parameters[i];
        CnSemSymbol *sym = cn_sem_scope_insert_symbol(function_scope,
                                   param->name,
                                   param->name_length,
                                   CN_SEM_SYMBOL_VARIABLE);
        if (sym) {
            sym->type = param->declared_type;
            sym->is_const = param->is_const;
        }
    }
    
    cn_sem_build_block_scope(function_scope, function_decl->body, diagnostics);
    // 【注意】不再释放 function_scope，它现在由 AST 节点拥有
}
```

#### 3. 修改 semantic_passes.c

```c
// src/semantics/checker/semantic_passes.c

bool cn_sem_resolve_names(CnSemScope *global_scope,
                          CnAstProgram *program,
                          CnDiagnostics *diagnostics)
{
    for (size_t i = 0; i < program->function_count; ++i) {
        CnAstFunctionDecl *fn = program->functions[i];
        
        // 【关键修改】使用 AST 节点中保存的作用域
        CnSemScope *fn_scope = fn->owning_scope;
        if (!fn_scope) {
            // 如果没有作用域（可能是函数原型），跳过
            continue;
        }
        
        resolve_block_names(fn_scope, fn->body, diagnostics);
        // 【注意】不再释放 fn_scope，它由 AST 节点拥有
    }
    
    return cn_support_diagnostics_error_count(diagnostics) == 0;
}
```

#### 4. 修改 irgen.c

```c
// src/ir/gen/irgen.c

void cn_ir_gen_function(CnIrGenContext *ctx, CnAstFunctionDecl *func, CnSemScope *parent_scope)
{
    // ...
    
    if (func->body) {
        // 【关键修改】使用 AST 节点中保存的作用域
        CnSemScope *func_scope = func->owning_scope;
        if (func_scope) {
            ctx->current_scope = func_scope;
        } else {
            // 如果没有作用域，创建新的（向后兼容）
            func_scope = cn_sem_scope_new(CN_SEM_SCOPE_FUNCTION, parent_scope);
            ctx->current_scope = func_scope;
        }
        
        cn_ir_gen_block(ctx, func->body);
        
        // 【注意】如果作用域来自 AST 节点，不释放
        if (!func->owning_scope) {
            cn_sem_scope_free(func_scope);
        }
        ctx->current_scope = ctx->global_scope;
    }
    
    // ...
}
```

#### 5. 修改 AST 释放逻辑

```c
// src/frontend/parser/ast_free.c

void cn_frontend_ast_function_free(CnAstFunctionDecl *function_decl)
{
    if (!function_decl) return;
    
    // 【新增】释放函数作用域
    if (function_decl->owning_scope) {
        cn_sem_scope_free(function_decl->owning_scope);
        function_decl->owning_scope = NULL;
    }
    
    // 释放其他资源...
}

void cn_frontend_ast_block_free(CnAstBlockStmt *block)
{
    if (!block) return;
    
    // 【新增】释放块作用域
    if (block->owning_scope) {
        cn_sem_scope_free(block->owning_scope);
        block->owning_scope = NULL;
    }
    
    // 释放其他资源...
}
```

## 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AST 节点内存增加 | 低 | 每个函数/块只增加一个指针 |
| 向后兼容性 | 中 | 保留 `owning_scope = NULL` 的处理逻辑 |
| 作用域生命周期管理 | 中 | 明确所有权：AST 节点拥有作用域 |
| 多线程安全 | 低 | 当前编译器是单线程的 |

---

## 预期效果

1. **类型信息不丢失**：`semantic_passes` 和 `irgen` 可以访问 `scope_builder` 创建的符号
2. **代码更简洁**：移除重复的作用域创建和释放代码
3. **架构更清晰**：作用域生命周期与 AST 节点绑定
4. **便于扩展**：为调试信息、IDE 支持等功能奠定基础

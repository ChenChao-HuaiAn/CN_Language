# 021 CN语言代码生成器重构方案

## 一、问题概述

CN语言自托管编译器在编译过程中，代码生成器存在以下三个主要问题：

1. **结构体和函数声明的输出顺序问题** - 结构体定义可能依赖其他结构体，当前没有拓扑排序
2. **同一函数被多次声明的问题** - 缺少声明去重机制
3. **指针成员访问的类型推断问题** - 指针类型的成员访问没有正确生成 `->` 运算符

---

## 二、问题一：结构体和函数声明输出顺序问题

### 2.1 问题分析

#### 当前实现位置
- 文件：[`src/backend/cgen/cgen.c`](src/backend/cgen/cgen.c)
- 函数：[`cn_cgen_module_with_structs_to_file()`](src/backend/cgen/cgen.c:1806)

#### 当前输出顺序
```
1. 枚举定义 (program->enums)
2. 全局结构体定义 (program->structs)
3. 局部结构体定义 (从函数体收集)
4. 接口定义 (program->interfaces)
5. 类定义 (program->classes)
6. 全局变量声明
7. 函数前向声明
8. 函数定义
```

#### 问题根源
1. **结构体间依赖未处理**：结构体A的字段可能是结构体B类型，如果B定义在A之后，C编译器会报错
2. **循环依赖未处理**：结构体A和B互相引用时，需要前向声明
3. **导入模块的类型定义顺序**：[`cn_cgen_import_forward_declarations()`](src/backend/cgen/cgen.c:2495) 已实现三阶段输出，但当前模块的结构体没有类似处理

#### 代码证据

```c
// cgen.c:1878-1883 - 当前直接按数组顺序输出结构体
if (program->struct_count > 0) {
    fprintf(file, "// CN Language Global Struct Definitions\n");
    for (size_t i = 0; i < program->struct_count; i++) {
        cn_cgen_struct_decl(&ctx, program->structs[i]);
    }
}
```

### 2.3 推荐方案

**推荐方案B**（两遍扫描），原因：
1. 实现简单，修改量小
2. C编译器会自动处理前向声明后的完整定义
3. 与现有的导入模块处理方式一致

## 三、问题二：同一函数被多次声明的问题

### 3.1 问题分析

#### 当前实现位置
- 文件：[`src/backend/cgen/cgen.c`](src/backend/cgen/cgen.c)
- 函数：[`cn_cgen_function_forward_declaration()`](src/backend/cgen/cgen.c:2443)

#### 问题根源
1. **选择性导入**：同一函数可能被多个模块选择性导入
2. **通配符导入**：多个模块的通配符导入可能包含相同函数
3. **当前模块与导入模块**：当前模块定义的函数可能与导入的函数同名

#### 代码证据

```c
// cgen.c:2699-2714 - 选择性导入直接输出声明，无去重

        CnSemSymbol *symbol = cn_sem_scope_lookup(global_scope, member->name, member->name_length);
        if (symbol && symbol->type) {
            if (symbol->kind == CN_SEM_SYMBOL_FUNCTION) {
                cn_cgen_function_forward_declaration(file, symbol_name, symbol_name_len, symbol->type);
                // 问题：没有检查是否已声明过
            }
        }
    }
}
```

### 3.2 修改方案

#### 方案：添加已声明函数集合

**修改位置**：[`src/backend/cgen/cgen.c`](src/backend/cgen/cgen.c)

**实现思路**：
1. 维护一个已声明函数名的哈希集合
2. 在输出声明前检查是否已声明
3. 声明后添加到集合

**代码片段**：

```c
// 新增：已声明函数名集合（与现有的 g_generated_type_names 类似）
#define MAX_DECLARED_FUNCTIONS 512
static const char *g_declared_function_names[MAX_DECLARED_FUNCTIONS];
static size_t g_declared_function_count = 0;

// 新增：检查函数是否已声明
static bool is_function_already_declared(const char *name, size_t name_len) {
    for (size_t i = 0; i < g_declared_function_count; i++) {
        if (strncmp(g_declared_function_names[i], name, name_len) == 0 &&
            g_declared_function_names[i][name_len] == '\0') {
            return true;
        }
    }
    return false;
}

// 新增：标记函数已声明
static void mark_function_as_declared(const char *name, size_t name_len) {
    if (g_declared_function_count >= MAX_DECLARED_FUNCTIONS) return;
    
    char *name_copy = malloc(name_len + 1);
    if (!name_copy) return;
    memcpy(name_copy, name, name_len);
    name_copy[name_len] = '\0';
    
    g_declared_function_names[g_declared_function_count++] = name_copy;
}

// 新增：重置已声明函数集合
static void reset_declared_functions(void) {
    for (size_t i = 0; i < g_declared_function_count; i++) {
        free((void*)g_declared_function_names[i]);
    }
    g_declared_function_count = 0;
}

// 修改：cn_cgen_function_forward_declaration 函数
static void cn_cgen_function_forward_declaration(FILE *file, const char *func_name, size_t func_name_len, CnType *func_type) {
    if (!file || !func_name || !func_type || func_type->kind != CN_TYPE_FUNCTION) {
        return;
    }
    
    // 新增：检查是否已声明
    if (is_function_already_declared(func_name, func_name_len)) {
        return;  // 跳过重复声明
    }
    
    // ... 原有的声明生成代码 ...
    
    // 新增：标记为已声明
    mark_function_as_declared(func_name, func_name_len);
}
```

2. [`cn_cgen_module_with_imports_to_file()`](src/backend/cgen/cgen.c:2751) - 同样需要重置

## 四、问题三：指针成员访问的类型推断问题

### 4.1 问题分析

#### 当前实现位置
- IR生成：[`src/ir/gen/irgen.c:1154`](src/ir/gen/irgen.c:1154) - `CN_AST_EXPR_MEMBER_ACCESS` 处理
- 代码生成：[`src/backend/cgen/cgen.c:408`](src/backend/cgen/cgen.c:408) - `CN_AST_EXPR_MEMBER_ACCESS` 处理

#### 问题根源

**CN语言语法**：指针成员访问使用 `.` 语法，而不是 C 语言的 `->`

```cn
// CN语言代码
结构体指针.成员 = 值;  // 正确
结构体指针->成员 = 值; // 语法错误
```

**期望生成的C代码**：
```c
struct_ptr->member = value;  // 正确
```

#### 代码证据

```c
// irgen.c:1198-1199 - 判断对象是否为指针类型
bool object_is_pointer = (expr->as.member.object->type &&
                          expr->as.member.object->type->kind == CN_TYPE_POINTER);

// irgen.c:1211-1215 - 设置类型信息到操作数
if (expr->as.member.object->type) {
    object_op.type = expr->as.member.object->type;
}

// cgen.c:448 - 代码生成时判断是否使用箭头
if (expr->as.member.is_arrow) {
    fprintf(ctx->output_file, "->");
} else {
    fprintf(ctx->output_file, ".");
}
```

**问题**：`is_arrow` 标志在解析时设置，但CN语言中指针访问不使用 `->` 语法，所以 `is_arrow` 始终为 `false`。

### 4.2 修改方案

#### 方案A：在代码生成器中根据类型判断（推荐）

**修改位置**：[`src/backend/cgen/cgen.c:408`](src/backend/cgen/cgen.c:408)

**实现思路**：
1. 检查对象类型是否为指针
2. 如果是指针，自动使用 `->`
3. 否则使用 `.`

**代码片段**：

```c
case CN_AST_EXPR_MEMBER_ACCESS:
    // 成员访问表达式：obj.member 或 ptr->member
    {
        // 检查是否为静态成员访问
        if (expr->as.member.is_static_member && expr->as.member.class_name) {
            // 静态成员访问：生成 类名_成员名
            fprintf(ctx->output_file, "%.*s_%.*s",
                    (int)expr->as.member.class_name_length, expr->as.member.class_name,
                    (int)expr->as.member.member_name_length, expr->as.member.member_name);
        }
        // 检查对象是否为自身指针
        else if (expr->as.member.object && expr->as.member.object->is_this_pointer) {
            fprintf(ctx->output_file, "self->%.*s",
                    (int)expr->as.member.member_name_length,
                    expr->as.member.member_name);
        }
        // 检查对象是否为基类指针
        else if (expr->as.member.object && expr->as.member.object->is_base_pointer) {
            // 基类访问：base->member
            if (ctx->current_class && ctx->current_class->base_count > 0) {
                CnInheritanceInfo *base_info = &ctx->current_class->bases[0];
                fprintf(ctx->output_file, "((struct %.*s*)((char*)self + %.*s_%.*s_OFFSET))->%.*s",
                        (int)base_info->base_class_name_length, base_info->base_class_name,
                        (int)ctx->current_class->name_length, ctx->current_class->name,
                        (int)base_info->base_class_name_length, base_info->base_class_name,
                        (int)expr->as.member.member_name_length,
                        expr->as.member.member_name);
            }
        } else {
            // 普通成员访问
            cn_cgen_expr_simple(ctx, expr->as.member.object);
            
            // 【修改点】根据对象类型判断使用 "." 还是 "->"
            // CN语言中，指针类型的成员访问使用 "." 语法，但生成C代码时需要转换为 "->"
            bool use_arrow = expr->as.member.is_arrow;  // 显式使用箭头语法
            
            // 【新增】检查对象类型是否为指针
            if (!use_arrow && expr->as.member.object && expr->as.member.object->type) {
                CnType *obj_type = expr->as.member.object->type;
                if (obj_type->kind == CN_TYPE_POINTER) {
                    use_arrow = true;  // 指针类型自动使用箭头
                }
            }
            
            if (use_arrow) {
                fprintf(ctx->output_file, "->");
            } else {
                fprintf(ctx->output_file, ".");
            }
            
            fprintf(ctx->output_file, "%.*s",
                    (int)expr->as.member.member_name_length,
                    expr->as.member.member_name);
        }
    }
    break;
```

## 五、实施优先级

### 5.1 优先级排序

| 优先级 | 问题 | 原因 |
|--------|------|------|
| **高** | 问题三：指针成员访问类型推断 | 影响所有使用指针的代码，直接导致编译失败 |
| **中** | 问题二：函数声明去重 | 影响多模块项目，可能导致C编译警告/错误 |
| **低** | 问题一：结构体输出顺序 | 大多数情况下结构体定义顺序正确，问题较少出现 |

### 5.2 实施步骤

1. **第一步**：修复指针成员访问类型推断问题

   - 测试指针成员访问代码

2. **第二步**：添加函数声明去重机制
   - 在 [`src/backend/cgen/cgen.c`](src/backend/cgen/cgen.c) 添加去重函数

## 七、风险评估

### 7.1 潜在风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 前向声明可能与某些C编译器不兼容 | 低 | 主流C编译器（GCC、Clang、MSVC）都支持 |
| 函数去重集合内存泄漏 | 中 | 确保在模块生成结束时释放内存 |
| 指针类型判断错误 | 高 | 添加类型检查断言和调试日志 |

### 7.2 回归测试

1. 现有测试用例全部通过

## 八、总结

本方案针对CN语言代码生成器的三个主要问题进行了详细分析，并提供了具体的修改方案：

1. **结构体输出顺序**：采用两遍扫描方案，先输出前向声明再输出完整定义
2. **函数声明去重**：添加已声明函数集合，在输出前检查去重
3. **指针成员访问**：在代码生成时根据对象类型自动选择 `.` 或 `->`

这些修改将显著提高代码生成器的健壮性，支持更复杂的程序结构。

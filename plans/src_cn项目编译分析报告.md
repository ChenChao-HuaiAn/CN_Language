## 根本原因分析

### 问题1：模块导入时符号类型信息丢失

从调试日志中发现关键问题：

```
[DEBUG] cn_sem_is_same_symbol: name=词元, kind=5, NO MATCH (type1=0000000000000000, type2=0000000000000000, ...)
```

**原因分析**：
1. `词元.cn` 模块定义了 `词元类型枚举` 枚举和 `词元` 结构体
2. 模块名 `词元` 与结构体名 `词元` 相同
3. 当其他文件导入 `词元.cn` 时，模块符号覆盖了结构体符号
4. 导致结构体符号的类型信息（`sym->type`）为 NULL

### 问题2：枚举类型识别失败

```
[DEBUG] MEMBER_ACCESS: looking up '词元类型枚举', found sym=..., kind=3, type=...
错误(17): 语义错误：成员访问操作的对象必须是结构体或枚举类型
```

**原因分析**：
- 枚举符号被正确找到（kind=3 表示枚举）
- 但类型推导时无法正确识别为枚举类型
- 可能是 `enum_scope` 未正确传递

### 问题3：函数符号未正确导入

```
错误(16): 语义错误：未定义的标识符 '创建类型节点'
错误(16): 语义错误：未定义的标识符 '解析类型'
错误(16): 语义错误：未定义的标识符 '解析表达式'
```

**原因分析**：
- 这些函数在其他模块中定义
- 导入时符号未被正确复制到当前作用域
- 可能是选择性导入逻辑的问题

## 修复结果（2026-04-11 20:11 更新）

### 问题根因

编译脚本逐个文件独立编译，每个文件启动独立进程，导致：
- 跨进程不共享模块缓存
- 模块重新编译时，旧的类型对象被释放
- 导入的符号引用无效指针，`sym->type` 为 NULL
- 类型检查失败，报告"成员访问操作的对象必须是结构体或枚举类型"

### 解决方案

修改 `tools/compile_all_src_cn.ps1` 编译脚本：
- **原方式**：逐个文件调用编译器（多进程）
- **新方式**：一次性传递所有文件给编译器（单进程）

### 修复后编译统计

|------|--------|--------|------|
| 总文件数 | 35 | 36 | +1 |

| 编译失败 | 12 | 0 | -12 |
| 成功率 | 65.7% | **100%** | +34.3% |

### 修复结论

问题已完全解决。根本原因不是编译器本身的符号导入逻辑问题，而是编译脚本的进程隔离导致模块缓存无法共享。使用单进程多文件编译方式后，所有文件均编译成功。

## 二、问题1：联合体语法不支持

### 2.1 问题详情

**错误特征**：
```
[DEBUG PARSER] 无法识别的token: kind=80, line=36, col=21
语法错误：缺少预期的标记
```

**根本原因**：
CN语言语法规范中没有定义 `联合体` 关键字。根据 [`plans/001 CN Language语法规范设计文档.md`](plans/001 CN Language语法规范设计文档.md) 第1.2节，CN语言只支持以下类型关键字：
- `整数`、`小数`、`字符串`、`布尔`、`空类型`、`结构体`、`枚举`

### 2.2 受影响文件

| 文件 | 联合体位置 | 用途 |
|------|------------|------|
| [`src_cn/诊断/诊断消息表.cn:36`](src_cn/诊断/诊断消息表.cn:36) | `联合体 参数值` | 存储不同类型参数的实际值 |
| [`src_cn/诊断/诊断恢复.cn`](src_cn/诊断/诊断恢复.cn) | 可能使用 | 恢复上下文相关 |
| [`src_cn/诊断/诊断修复.cn`](src_cn/诊断/诊断修复.cn) | 可能使用 | 修复建议相关 |

### 2.3 修复方案

#### 方案A：改为结构体（推荐）

**优点**：简单直接，无需修改编译器
**缺点**：内存占用增加

**修改示例**：

```cn
// 原代码（第36-40行）
联合体 参数值 {
    常量字符串 字符串值    // 字符串值
    整数 整数值            // 整数值
    整数 标记值            // Token类型值（用整数表示）
}

// 修改后
结构体 参数值 {
    常量字符串 字符串值    // 字符串值
    整数 整数值            // 整数值
    整数 标记值            // Token类型值
}
```

#### 方案B：使用类型标签+独立字段

**优点**：类型安全，可区分实际存储的类型
**缺点**：需要修改访问逻辑

**修改示例**：

```cn
/// 参数类型标签枚举
枚举 参数类型标签 {
    无值,        // 未设置
    字符串类型,  // 字符串值
    整数类型,    // 整数值
    标记类型     // Token类型值
}

/// 参数值结构体（带类型标签）
结构体 参数值 {
    参数类型标签 类型标签    // 类型标签
    常量字符串 字符串值      // 字符串值
    整数 整数值              // 整数值
    整数 标记值              // Token类型值
}
```

### 2.4 修复步骤

1. **修改 `src_cn/诊断/诊断消息表.cn`**
   - 将第36行的 `联合体 参数值` 改为 `结构体 参数值`
   - 检查所有使用 `参数值` 的地方，确保访问方式正确

2. **修改 `src_cn/诊断/诊断恢复.cn`**
   - 搜索文件中是否有 `联合体` 关键字
   - 如有，按相同方式修改

3. **修改 `src_cn/诊断/诊断修复.cn`**
   - 搜索文件中是否有 `联合体` 关键字
   - 如有，按相同方式修改

4. **验证修复**
   - 重新编译诊断模块的3个文件
   - 确认编译成功

## 三、问题2：模块符号解析问题

### 3.1 问题详情

**错误特征**：
```
语义错误：未定义的标识符 '当前词元'
语义错误：未定义的标识符 '前进'
语义错误：未定义的标识符 '匹配'
```

**根本原因**：
1. 模块导入后符号未正确解析
2. 作用域链可能存在断裂
3. 导出的符号未正确添加到当前作用域

### 3.2 受影响模块

| 模块 | 文件数 | 主要错误类型 |
|------|--------|--------------|
| 语法分析器 | 11个 | 未定义标识符 |
| 语义分析器 | 4个 | 未定义标识符、参数类型不匹配 |
| 模块加载器 | 1个 | 未定义标识符 |

### 3.3 问题分析

根据 [`src_cn/语法分析器/解析器.cn`](src_cn/语法分析器/解析器.cn) 的导入语句：

```cn
导入 ./解析器类型;           // 解析器结构体定义
导入 ../词法分析器/词元;     // 词元类型枚举定义
导入 ../诊断/诊断核心;       // 诊断系统
导入 ./解析器/类型解析;
导入 ./解析器/表达式解析;
导入 ./解析器/语句解析;
导入 ./解析器/声明解析;
```

问题可能出在：
1. **导入语句位置**：导入语句应该在文件顶部，公开/私有标签之前
2. **符号导出**：被导入模块可能没有正确标记 `公开:` 标签
3. **作用域链**：模块导入后，符号可能没有正确添加到当前作用域

### 3.4 修复方案

#### 方案A：检查并修正导入语句位置

确保所有导入语句位于：
1. 文件头部注释之后
2. 任何 `公开:` 或 `私有:` 标签之前

**错误示例**：
```cn
公开:

导入 ./模块;  // ❌ 错误：导入语句在公开标签之后
```

**正确示例**：
```cn
导入 ./模块;  // ✅ 正确：导入语句在公开标签之前

公开:
```

#### 方案B：确保符号正确导出

被导入模块必须：
1. 使用 `公开:` 标签导出需要外部访问的符号
2. 导出的函数、类型、枚举必须在 `公开:` 标签之后定义

**示例**：
```cn
// 词元.cn
公开:

枚举 词元类型 {
    // ...
}

函数 下一个词元(扫描器* 实例) -> 词元 {
    // ...
}
```

#### 方案C：检查编译器模块导入处理

需要检查编译器源码中的模块导入处理逻辑：

1. **语义分析器**：[`src/frontend/semantics/semantics.c`](src/frontend/semantics/semantics.c)
   - 检查 `cn_sem_process_import` 函数
   - 确保导入的符号正确添加到当前作用域

2. **模块加载器**：[`src/frontend/module_loader/module_loader.c`](src/frontend/module_loader/module_loader.c)
   - 检查模块符号导出逻辑
   - 确保公开符号正确导出

### 3.5 修复步骤

1. **检查所有失败文件的导入语句位置**
   - 确保导入语句在 `公开:` 标签之前
   - 列出需要修正的文件

2. **检查被导入模块的符号导出**
   - 确认 `词元.cn`、`诊断核心.cn` 等模块正确导出符号
   - 确认 `公开:` 标签位置正确

3. **检查编译器模块导入处理**（如需要）
   - 如果上述修改无效，检查编译器源码
   - 可能需要修复语义分析器的符号解析逻辑

4. **验证修复**
   - 重新编译受影响的模块
   - 确认符号解析错误消失

## 四、问题3：arg_kind=20 类型问题

### 4.1 问题详情

**错误特征**：
```
语义错误：函数 '追加整数' 参数 1 类型不匹配 (arg_kind=20, param_kind=1)
```

**根本原因**：
`arg_kind=20` 表示 `CN_TYPE_UNKNOWN`（未知类型），类型系统在参数传递时未能正确处理未知类型。

### 4.2 受影响模块

| 模块 | 文件数 | 主要错误类型 |
|------|--------|--------------|
| 代码生成器 | 2个 | 参数类型不匹配 |
| IR生成器 | 部分文件 | 参数类型不匹配 |

### 4.3 类型系统分析

根据CN语言类型定义（[`include/cnlang/frontend/ast.h`](include/cnlang/frontend/ast.h)）：

```c
typedef enum cn_type_kind {
    CN_TYPE_VOID = 0,      // 空类型
    CN_TYPE_INT = 1,       // 整数
    CN_TYPE_FLOAT = 2,     // 小数
    CN_TYPE_STRING = 3,    // 字符串
    CN_TYPE_BOOL = 4,      // 布尔
    // ...
    CN_TYPE_UNKNOWN = 20   // 未知类型
} cn_type_kind;
```

`kind=20` 表示类型系统无法确定表达式的类型，可能原因：
1. 类型推导失败
2. 表达式类型未正确设置
3. 模块导入时类型信息丢失

### 4.4 修复方案

#### 方案A：增强类型推导

在语义分析阶段，确保所有表达式都有正确的类型：

1. **检查表达式类型设置**
   - 函数调用表达式：返回类型应从函数声明获取
   - 变量引用：类型应从符号表获取
   - 字面量：类型应根据字面量值确定

2. **检查类型传播**
   - 赋值表达式：类型应从右侧表达式传播
   - 返回语句：类型应从返回表达式获取

#### 方案B：增强类型兼容性检查

在类型检查时，对 `CN_TYPE_UNKNOWN` 进行特殊处理：

1. **延迟类型检查**
   - 如果参数类型是 `CN_TYPE_UNKNOWN`，尝试从函数参数类型推断

2. **类型推断**
   - 如果无法确定类型，使用函数参数类型作为期望类型

### 4.5 修复步骤

1. **定位具体错误位置**
   - 查找 `追加整数` 函数调用位置
   - 分析参数类型为何是 `CN_TYPE_UNKNOWN`

2. **检查类型推导逻辑**
   - 检查语义分析器中的类型推导代码
   - 确保表达式类型正确设置

3. **增强类型兼容性检查**（如需要）
   - 修改类型检查逻辑
   - 对 `CN_TYPE_UNKNOWN` 进行特殊处理

4. **验证修复**
   - 重新编译受影响的模块
   - 确认类型不匹配错误消失

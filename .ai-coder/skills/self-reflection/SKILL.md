# 自我反思与学习 (Self-Reflection)

让Agent能够分析自己的输出，识别错误和不足，并从错误中学习改进。

## 何时使用此技能

使用此技能当：

- 任务完成后需要反思执行过程和结果
- 用户指出错误时需要进行深度分析
- 定期回顾历史输出质量
- 代码审查后需要自我改进
- 发现输出不符合预期时
- 需要从失败中学习教训

## 核心能力

| 能力 | 描述 |
|------|------|
| 输出分析 | 分析已完成的任务输出，识别潜在问题 |
| 错误识别 | 检测逻辑错误、遗漏、不一致等问题 |
| 根因分析 | 追溯问题产生的根本原因 |
| 改进建议 | 生成具体的改进措施和最佳实践 |
| 学习记录 | 将学到的教训持久化存储 |

## 使用方法

### 1. 任务完成后的反思

当完成一个任务后，执行以下反思流程：

```markdown
## 反思检查清单

### 输出质量检查
- [ ] 输出是否完整回答了用户的问题？
- [ ] 输出是否准确无误？
- [ ] 输出是否清晰易懂？
- [ ] 输出是否遵循了最佳实践？

### 过程效率检查
- [ ] 是否采用了最优的方法？
- [ ] 是否有不必要的步骤？
- [ ] 是否可以并行处理某些任务？

### 知识积累检查
- [ ] 是否学到了新的知识？
- [ ] 是否发现了需要改进的地方？
- [ ] 是否有可以复用的模式？
```

### 2. 错误分析流程

当发现错误时，按以下步骤分析：

```markdown
## 错误分析模板

### 错误描述
- **错误类型**: [逻辑错误/遗漏/不一致/性能问题/其他]
- **严重程度**: [高/中/低]
- **影响范围**: [描述影响范围]

### 根因分析
- **直接原因**: [导致错误的直接原因]
- **深层原因**: [导致错误的根本原因]
- **触发条件**: [什么条件下会触发此错误]

### 修复方案
- **立即修复**: [如何立即修复此错误]
- **预防措施**: [如何防止类似错误再次发生]
- **改进建议**: [相关的改进建议]

### 教训记录
- **学到了什么**: [从这次错误中学到了什么]
- **适用场景**: [这个教训适用于哪些场景]
- **关联知识**: [相关的知识点]
```

### 3. 反思结果输出格式

```yaml
# 反思结果数据结构
reflection_result:
  task_id: "任务唯一标识"
  timestamp: "2024-01-15T10:30:00Z"
  
  # 问题识别
  issues:
    - type: "error|warning|improvement"  # 问题类型
      severity: "high|medium|low"         # 严重程度
      description: "问题描述"
      location: "问题位置（文件、代码行等）"
      root_cause: "根本原因分析"
      
  # 改进建议
  improvements:
    - category: "knowledge|capability|performance"  # 改进类别
      action: "具体行动建议"
      priority: "high|medium|low"
      related_skill: "关联技能名称"  # 可选，指向其他技能
      
  # 学到的教训
  lessons_learned:
    - insight: "洞察内容"
      context: "上下文说明"
      applicable_scenarios: "适用场景"
```

## 与其他技能的协作

### 协作触发条件

| 触发场景 | 协作技能 | 协作方式 |
|----------|----------|----------|
| 发现知识缺口 | knowledge-accumulation | 将反思结果存储为知识 |
| 发现效率问题 | performance-optimization | 提交优化建议 |
| 发现能力不足 | capability-extension | 请求安装新技能 |

### 协作示例

```markdown
## 反思发现知识缺口

当反思发现缺少某方面知识时：

1. 生成知识存储请求：
   ```yaml
   knowledge_request:
     action: "store"
     store:
       category: "lesson"
       content: "学到的教训内容"
       context: "获取上下文"
       tags: ["标签1", "标签2"]
       confidence: 0.8
   ```

2. 调用 knowledge-accumulation 技能存储知识

## 反思发现效率问题

当反思发现工作流程效率低下时：

1. 生成优化请求：
   ```yaml
   optimization_request:
     action: "optimize"
     optimize:
       current_workflow: "当前工作流程描述"
       constraints: ["约束条件"]
       goals: ["优化目标"]
   ```

2. 调用 performance-optimization 技能分析优化

## 反思发现能力不足

当反思发现缺少必要能力时：

1. 生成能力请求：
   ```yaml
   capability_request:
     action: "identify"
     identify:
       current_task: "当前任务描述"
       missing_capabilities: ["缺少的能力"]
       context: "任务上下文"
   ```

2. 调用 capability-extension 技能搜索安装
```

## 提示模板

### 反思提示模板

使用 `prompts/reflect.md` 中的模板进行反思：

```markdown
# 反思提示

请对以下任务执行过程进行反思：

## 任务描述
{task_description}

## 执行过程
{execution_process}

## 输出结果
{output_result}

请从以下维度进行反思：

1. **准确性**: 输出是否准确？有无错误？
2. **完整性**: 是否完整回答了问题？有无遗漏？
3. **效率**: 执行过程是否高效？有无优化空间？
4. **可维护性**: 输出是否易于维护和扩展？

请输出反思结果，包括：
- 发现的问题
- 改进建议
- 学到的教训
```

### 错误分析模板

使用 `prompts/error-analysis.md` 中的模板分析错误：

```markdown
# 错误分析提示

请分析以下错误：

## 错误信息
{error_info}

## 上下文
{context}

## 预期行为
{expected_behavior}

## 实际行为
{actual_behavior}

请进行根因分析：

1. **直接原因**: 什么直接导致了这个错误？
2. **深层原因**: 根本原因是什么？
3. **触发条件**: 在什么条件下会触发此错误？
4. **修复方案**: 如何修复？
5. **预防措施**: 如何防止类似错误？
```

## 注意事项

1. **反思时机**: 
   - 任务完成后立即反思效果最佳
   - 不要在任务执行过程中打断进行反思

2. **反思深度**:
   - 不要只停留在表面问题
   - 追问"为什么"直到找到根本原因

3. **客观性**:
   - 保持客观，不要为错误找借口
   - 承认错误是学习的机会

4. **可操作性**:
   - 改进建议要具体、可执行
   - 避免模糊的建议如"提高质量"

5. **知识存储**:
   - 反思结果要及时存储到知识库
   - 使用 knowledge-accumulation 技能进行持久化

## 示例用法

### 示例1：代码生成后的反思

```markdown
## 反思结果

### 任务
为用户生成一个用户认证模块

### 发现的问题
1. **遗漏**: 没有处理密码重置功能
2. **不一致**: 错误消息格式不统一
3. **性能**: 数据库查询没有优化

### 改进建议
1. 添加密码重置功能
2. 统一错误消息格式，使用常量定义
3. 添加数据库索引，使用查询缓存

### 学到的教训
- 认证模块需要考虑完整的功能列表
- 错误消息格式应该在项目开始时统一
- 数据库查询优化应该在设计阶段考虑
```

### 示例2：错误分析后的反思

```markdown
## 错误分析结果

### 错误描述
生成的代码在生产环境抛出空指针异常

### 根因分析
1. **直接原因**: 没有对用户输入进行空值检查
2. **深层原因**: 没有遵循防御性编程原则
3. **触发条件**: 用户输入为空字符串时触发

### 修复方案
1. 添加输入验证
2. 使用 Optional 类型处理可能为空的值
3. 添加单元测试覆盖边界情况

### 教训记录
- **洞察**: 防御性编程是必要的，不能假设输入总是有效
- **适用场景**: 所有处理外部输入的代码
- **关联知识**: 防御性编程、输入验证、单元测试

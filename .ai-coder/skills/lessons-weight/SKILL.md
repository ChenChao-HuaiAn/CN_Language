# lessons.md 权重管理技能

## 概述

本技能为 AI Agent 提供 lessons.md 的权重管理能力，确保 AI 在任务开始时优先关注高频、高影响的历史问题，避免重复犯错。

---

## 何时使用此技能 / When to Use

**强制触发场景：**
1. **任务开始时** - 读取 lessons.md 必须应用权重规则
2. **记录错误时** - 写入错误记录必须计算权重
3. **发现重复错误** - 相同类型错误出现时必须检查历史

**不要使用：**
- 与错误记录无关的任务
- 纯代码编写任务（无错误发生）

---

## 核心规则

### 一、权重计算规则

**公式**：
```
权重 = 基础权重 × 类型系数 × 新鲜度系数 × 解决状态系数
```

**基础权重表**：

| 问题类型 | 基础权重 | 说明 |
|---------|---------|------|
| 设计缺陷 | 10 | 架构或设计层面问题 |
| 任务未完成 | 9 | 任务被遗漏或未完成 |
| 逻辑错误 | 8 | 代码逻辑错误 |
| 集成问题 | 7 | 模块间集成问题 |
| 配置错误 | 6 | 配置问题 |

**注意**：已移除"工具执行错误"类型，因为该类型错误价值低，AI无法从中学习。

**类型系数**：

| 内容特征 | 系数 |
|---------|------|
| 有详细分析 | ×2.0 |
| 有解决方案 | ×1.5 |
| 有预防措施 | ×1.3 |
| 模板化内容 | ×0.3 |

**新鲜度系数**：
```
新鲜度 = 1 / (1 + log10(天数 + 1))
```

**解决状态系数**：

| 状态 | 系数 |
|------|------|
| 未解决 | ×2.0 |
| 部分解决 | ×1.2 |
| 已解决 | ×1.0 |

### 二、读取优先级规则

**AI 读取 lessons.md 时必须遵循以下顺序：**

```markdown
## 读取流程

Step 1: 读取"⚠️ 高权重问题"部分
        └── 重点关注权重 ≥ 15 的问题（🔴 红色标记）
        └── 其次关注权重 10-14 的问题（🟡 黄色标记）

Step 2: 检查"问题类型统计"
        └── 了解各类问题分布
        └── 识别高频问题类型

Step 3: 根据当前任务类型，匹配相关历史问题
        └── 代码编写 → 查找"逻辑错误"、"设计缺陷"
        └── 配置修改 → 查找"配置错误"
        └── 集成开发 → 查找"集成问题"

Step 4: 低权重历史记录仅在需要时查阅
        └── 权重 < 5 的内容可跳过
```

### 三、写入规范规则

#### 3.1 错误记录质量检查

**记录错误前必须通过以下质量检查：**

```markdown
## 质量检查清单

### 必须包含的信息（缺一不可）
- [ ] **场景**：具体任务、操作步骤
- [ ] **代码/命令**：具体的代码片段或命令
- [ ] **错误信息**：实际的错误输出或异常
- [ ] **根因分析**：技术层面的原因
- [ ] **具体解决方案**：具体的代码修改、配置变更
- [ ] **预防措施**：具体检查步骤

### 质量评分
- 6项全部满足 → 高质量记录，允许写入
- 4-5项满足 → 中等质量，建议补充后写入
- 少于4项 → 低质量记录，禁止写入
```

#### 3.2 低价值记录自动过滤

**以下类型的错误禁止记录：**

| 禁止类型 | 示例 | 原因 |
|---------|------|------|
| 模糊的"工具执行错误" | `[RooCode#say] task xxx aborted` | 没有具体场景、代码、错误信息 |
| 没有具体代码/命令 | "某个命令执行失败" | 无法复现，无法预防 |
| 没有具体解决方案 | "检查工具依赖项、查看日志" | 过于笼统，没有指导意义 |
| 临时性、不可复现错误 | 网络超时、服务暂时不可用 | 不是AI可以预防的问题 |

**过滤规则**：
```
IF 错误类型 == "工具执行错误" AND 没有具体代码/命令:
    拒绝记录，返回提示："此错误缺少具体信息，无法记录"
    
IF 描述包含 "aborted" OR "timeout" OR "临时":
    拒绝记录，返回提示："临时性错误不适合记录到 lessons.md"
```

#### 3.3 具体信息完整性验证

**验证函数**：
```python
def validate_error_record(record):
    """验证错误记录的完整性"""
    required_fields = {
        '场景': lambda x: len(x) > 20,  # 至少20字符
        '代码/命令': lambda x: '```' in x or len(x) > 10,  # 有代码块或具体命令
        '错误信息': lambda x: len(x) > 10,  # 至少10字符
        '根因分析': lambda x: len(x) > 15,  # 至少15字符
        '解决方案': lambda x: len(x) > 20 and '检查' not in x[:10],  # 具体方案，不是"检查xxx"
        '预防措施': lambda x: len(x) > 10  # 至少10字符
    }
    
    missing = []
    for field, validator in required_fields.items():
        if field not in record or not validator(record.get(field, '')):
            missing.append(field)
    
    return len(missing) == 0, missing
```

### 四、清理维护规则

**定期执行（建议每周）：**

1. **合并重复错误**
   - 同一天内相同错误 → 合并为一条，记录次数
   - 跨天相同错误 → 保留首末记录

2. **压缩低价值内容**
   - 权重 < 3 的模板化错误 → 删除
   - 超过 30 天的低质量记录 → 删除

3. **更新权重摘要**
   - 重新计算所有记录权重
   - 更新高权重问题列表

## 检查清单

### 任务开始前检查

```markdown
## lessons.md 读取检查

- [ ] 已读取"⚠️ 高权重问题"部分
- [ ] 已关注权重 ≥ 15 的问题
- [ ] 已检查当前任务相关的历史问题

```markdown
## 错误记录检查

### 质量检查
- [ ] 有具体场景描述
- [ ] 有具体代码/命令
- [ ] 有实际错误信息
- [ ] 有技术层面的根因分析

# 错误记录写入提示

## 强制指令

当需要记录错误到 lessons.md 时，你**必须**遵循以下规则：

### 1. 选择正确的问题类型

从以下类型中选择**最准确**的一个：

| 类型 | 基础权重 | 适用场景 |
|------|---------|---------|
| 设计缺陷 | 10 | 架构设计问题、模块设计问题、系统设计缺陷 |
| 任务未完成 | 9 | 任务遗漏、未完成标记完成、计划未执行 |
| 逻辑错误 | 8 | 代码逻辑错误、算法错误、条件判断错误 |
| 集成问题 | 7 | 模块集成、API 对接问题、依赖冲突 |
| 配置错误 | 6 | 配置项错误、环境配置问题、参数设置错误 |
| 工具执行错误 | 2 | 命令执行失败、工具异常、系统错误 |
| 其他 | 3 | 无法归类的其他问题 |

### 2. 记录格式规范

```markdown
- [YYYY-MM-DD HH:mm] **问题类型**: [类型]
  - **描述**: [简明扼要描述问题，不超过50字]
  - **原因**: [根本原因分析，推荐填写]
  - **解决**: [解决方案，推荐填写]
  - **预防**: [预防措施，推荐填写]
```

### 3. 权重计算

写入前计算权重：

```
权重 = 基础权重 × 类型系数

类型系数计算：
- 有原因分析：×2.0
- 有解决方案：×1.5
- 有预防措施：×1.3
- 无分析无方案：×0.5
```

**示例计算**：
- 设计缺陷 + 原因 + 解决 + 预防 = 10 × 2.0 × 1.5 × 1.3 = 39
- 设计缺陷 + 原因 + 解决 = 10 × 2.0 × 1.5 = 30
- 设计缺陷 + 原因 = 10 × 2.0 = 20
- 设计缺陷（无分析）= 10 × 0.5 = 5
- 工具执行错误（模板）= 2 × 0.3 = 0.6 ≈ 1

### 4. 禁止行为

- ❌ 禁止写入重复的错误（检查是否已存在）
- ❌ 禁止只写描述不写原因（降低权重）
- ❌ 禁止使用模板化内容填充
- ❌ 禁止遗漏高权重问题的记录

## 写入确认模板

写入完成后，输出：

```markdown
✅ **错误记录已写入 lessons.md**

- 问题类型：[类型]
- 权重：[N]
- 记录位置：lessons.md 第 [N] 行
- 是否需要关注：[是/否，权重≥10时为是]
```

## 示例

### 示例 1：设计缺陷记录

```markdown
- [2026-03-17 23:00] **问题类型**: 设计缺陷
  - **描述**: 记忆系统存储与内存优化方案阶段二未实施
  - **原因**: 阶段一基础设施已完成，但阶段二核心改造未执行，导致组件独立存在但未整合
  - **解决**: 重构MemoryStore使用适配器模式，实现延迟加载，添加自动迁移检测
  - **预防**: 每个阶段完成后进行集成验证，确保组件正确整合
```

权重计算：10 × 2.0 × 1.5 × 1.3 = **39**

### 示例 2：任务未完成记录

```markdown
- [2026-03-17 23:05] **问题类型**: 任务未完成
  - **描述**: 插件打包任务未完成就声称完成
  - **原因**: 未验证打包结果，直接声称任务完成
  - **解决**: 添加打包验证步骤，确认VSIX文件存在且大小正常
```

权重计算：9 × 2.0 × 1.5 = **27**

### 示例 3：工具执行错误记录（低权重）

```markdown
- [2026-03-17 23:10] **问题类型**: 工具执行错误
  - **描述**: [RooCode#say] task xxx aborted
```

权重计算：2 × 0.5 = **1**（低权重，可压缩）

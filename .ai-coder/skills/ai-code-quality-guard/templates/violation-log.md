# 违规记录模板

> 此模板用于记录代码编写过程中的违规行为，支持经验沉淀和持续改进

---

## 违规记录格式

每条违规记录包含以下字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| **记录时间** | 发现违规的时间 | 2026-03-13 14:30 |
| **任务ID** | 关联的任务ID | TASK-123 |
| **违规类型** | 违规的分类 | TODO占位/空方法体/魔法值/参数缺失 |
| **严重程度** | 违规的严重程度 | 严重/警告/提示 |
| **文件路径** | 违规所在的文件 | src/user/service.ts |
| **行号** | 违规所在的行号 | 45 |
| **违规描述** | 具体的违规内容 | 发现TODO注释: "TODO: implement validation" |
| **修正方式** | 如何修正该违规 | 移除TODO，实现完整的验证逻辑 |
| **修正状态** | 是否已修正 | 已修正/待修正/无法修正 |

## 经验沉淀

### 常见违规模式

#### 模式 1: TODO占位

**典型场景**：
- 复杂逻辑暂时不知道如何实现
- 依赖其他未完成的接口
- 时间紧迫，先留占位符

**预防措施**：
- 任务拆分时明确每个子任务的实现细节
- 提供足够的上下文和示例代码
- 预检阶段识别潜在的实现难点

#### 模式 2: 空方法体

**典型场景**：
- 接口定义后忘记实现
- 不确定方法应该做什么
- 方法职责划分不清晰

**预防措施**：
- 任务描述中明确方法职责
- 提供接口文档或示例实现
- 禁区检测阶段自动拦截

#### 模式 3: 魔法值

**典型场景**：
- 快速实现时直接写死数值
- 不确定是否需要配置化
- 复制代码时未重构

**预防措施**：
- 编码规范强调禁止魔法值
- 提供常量定义文件模板
- 代码审查时重点关注

## 集成说明

### 与 lessons.md 的集成

违规记录应同步写入项目的 `lessons.md` 文件：

```markdown
## [YYYY-MM-DD] 代码质量违规记录

### 违规摘要
- 任务ID: TASK-XXX
- 违规数量: X 个
- 严重程度: 严重 X 个 / 警告 X 个

### 经验教训
1. [从违规中总结的经验]
2. [预防类似违规的建议]
```

### 与 ai-error-record 技能的集成

检测到严重违规时，调用 `ai-error-record` 技能：

```
skill(skill="ai-error-record", args="error_type=代码质量违规; error_description=[违规描述]; root_cause=[违规原因]; solution=[修正方案]")
```

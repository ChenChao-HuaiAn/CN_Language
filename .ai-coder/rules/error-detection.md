# 错误自动检测和记录规则

## 重要说明
**Ai 规则系统的局限性**：本规则文件是 AI 行为的指导意见，但 Ai 规则系统**无法真正拦截工具执行结果**。错误的自动检测和记录依赖于 AI 的**自我监控**和**自觉遵守**规则。

## 触发条件
当 Ai 执行命令或工具调用出现以下情况时，**AI 应该**自动触发错误记录：

1. **命令执行失败**：`execute_command` 工具返回非零退出码
2. **文件操作错误**：`read_file`、`write_to_file`、`apply_diff` 等工具操作失败
3. **语法错误**：用户提到"语法错误"、"命令错误"、"执行失败"等关键词
4. **工具调用异常**：任何工具调用返回错误信息

---

## 错误记录前的上下文收集（强制）

AI 在调用 ai-error-record 技能前，必须执行以下步骤：

### Step 1: 收集完整上下文
```
必须收集：
- 具体执行的命令或工具名称
- 完整的命令参数或工具调用参数
- 实际的错误输出（错误消息、堆栈跟踪）
- 涉及的文件路径（如有）
```

### Step 2: 分析根本原因
```
禁止使用模板化分析，必须：
- 根据具体错误输出分析原因
- 结合项目上下文分析原因
- 如果无法确定原因，写"待分析"而非模板
```

### Step 3: 提供针对性解决方案
```
禁止使用通用建议，必须：
- 提供针对具体错误的解决步骤
- 如果无法提供方案，写"待调查"而非模板
```

### Step 4: 判断是否值得记录
```
以下情况不应记录（低价值）：
- 临时性网络错误
- 用户主动取消操作
- 无法获取具体错误信息的抽象错误
- task aborted 类错误（无具体错误输出）
```

---

## AI 自我监控要求
AI 必须在每次工具调用后：

1. **检查结果**：确认工具调用是否成功
2. **识别错误**：当检测到错误时，分析错误类型、描述、原因
3. **主动记录**：调用 `skill(skill="ai-error-record", args="...")` 记录错误
4. **诚实报告**：不得在未完成任务时声称"任务完成"

---

## 在响应生成前的最终检查
在调用 `attempt_completion` 之前，AI 必须执行以下检查：

1. **确认所有子任务已完成**
2. **确认所有测试都通过**
3. **确认没有未处理的错误**
4. **确认用户没有未解决的反馈**

**如果存在未完成的子任务或错误，不得调用 attempt_completion，不得声称"任务完成"**

## 使用示例

### 命令执行失败后的正确操作
```python
# 执行命令后，AI 应该检查结果
# 如果命令失败（退出码非 0），AI 应该：
skill(
    skill="ai-error-record",
    args="error_type=命令执行错误; error_description=执行 'pnpm build' 失败，错误输出：'Module not found: ./types'，参数：--filter=webview-ui; root_cause=webview-ui 缺少 types 目录，导致构建时模块解析失败; solution=创建 webview-ui/src/types 目录并添加缺失的类型定义文件"
)
```

### 文件操作错误后的正确操作
```python
# 当文件操作失败时，AI 应该：
skill(
    skill="ai-error-record",
    args="error_type=文件操作错误; error_description=调用 read_file 工具失败，参数 path='./src/missing.ts'，错误输出：ENOENT: no such file or directory; root_cause=文件路径错误，missing.ts 文件不存在于 src 目录; solution=检查文件路径拼写，确认文件是否已被删除或移动"
)
```

### ❌ 错误示例（禁止使用）
```python
# 以下调用方式是错误的，禁止使用：
skill(
    skill="ai-error-record",
    args="error_type=命令执行错误; error_description=task-123 失败; root_cause=工具内部逻辑错误或依赖项问题; solution=检查工具依赖项是否正确安装"
)
# 问题：error_description 只记录 task ID，root_cause 和 solution 都是模板化内容
```

## 注意事项
1. **避免重复记录**：同一错误只记录一次
2. **详细描述**：错误描述必须包含具体命令/工具、参数、错误输出
3. **具体分析**：根本原因分析必须针对具体问题，禁止模板化
4. **可操作方案**：解决方案必须是可执行的具体步骤
5. **不影响主流程**：错误记录不应中断主要任务流程
6. **诚实报告**：在未完成任务或测试失败时，不得声称"任务完成"

---

## 已知局限性
- 规则系统依赖 AI 自觉遵守，没有强制触发机制
- AI 需要主动检查结果并调用技能，不会自动拦截
- 用户需要监督 AI 是否遵守了规则

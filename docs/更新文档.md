
### 当前思路小结

- 已梳理现有实现：  
  - 语法层面：`导入` 语句已经区分了**路径导入**（`module_path`，如 `导入 ./工具;`）与**传统导入**（`module_name`，如 `导入 工具;`）。  
  - 模块加载器：`cn_module_loader_resolve_path` 已支持从搜索路径查找 `模块名.cn`，并在失败时再查找 `包目录/__包__.cn`。  
  - 相对路径加载：`cn_module_loader_load_relative` 已能从当前文件所在目录出发，构造真实路径，并在文件不存在时尝试 `module_path/__包__.cn`。  
  - 语义分析：`cn_sem_build_scopes_with_loader` 对 **路径导入** 已经接入模块加载器（相对/绝对路径、`从...导入` 与 `导入 路径` 皆有逻辑），但对 **`导入 模块名;`** 目前仍只在同一文件内查找内联模块，尚未使用模块加载器。

下面给出**具体改造方案**，并说明你可以如何在本地按步骤实现这些修改。

### 一、在模块加载器中更清晰地区分模块文件与包目录

目标：  
- **相对路径导入**（如 `导入 ./模块1;` 或 `从 ./工具 导入 {...};`）时，优先把路径当作**包目录**（查找 `__包__.cn`），失败时再当作**单文件模块**（查找 `模块.cn`）。  
- 为相对路径加载得到的 `CnModuleMetadata` 正确设置 `type`（`CN_MODULE_TYPE_FILE` 或 `CN_MODULE_TYPE_PACKAGE`），与 `cn_module_loader_load` 的行为保持一致。  

当前实现（简化后）在 [`module_loader.c`](file:///c:/Users/ChenChao/Documents/gitcode/CN_Language/src/frontend/module_loader/module_loader.c#L1457-L1588) 中大致是：

```c
// 构建 module_path = base_dir/segment1/segment2/...
// 然后先尝试 module_path.cn，再 fallback 到 module_path/__包__.cn

sprintf(file_path, "%s.cn", module_path);

if (!file_exists(file_path)) {
    free(file_path);
    char *package_init_path = path_join(module_path, "__包__.cn");
    free(module_path);

    if (!package_init_path || !file_exists(package_init_path)) {
        free(package_init_path);
        return NULL;
    }

    file_path = package_init_path;
} else {
    free(module_path);
}

// 后面创建 module_id / metadata，并固定写死 metadata->type = CN_MODULE_TYPE_FILE;
```

#### 1. 调整查找顺序：优先识别包

**建议修改方式：**

在 `cn_module_loader_load_relative` 中，构建完 `module_path` 之后，使用如下顺序：

1. **先尝试作为包目录**：`module_path/__包__.cn`  
2. 如果不存在包初始化文件，再尝试 `.cn` 文件：`module_path.cn`  

伪代码示意（替换原有 “添加 .cn 后缀 + fallback __包__.cn” 区块）：

```c
// module_path 已是 base_dir/segment1/.../segmentN

CnModuleType module_type = CN_MODULE_TYPE_FILE;
char *file_path = NULL;

// 1. 优先尝试 package init: module_path/__包__.cn
char *package_init_path = path_join(module_path, CN_PACKAGE_INIT_FILENAME);
if (package_init_path && file_exists(package_init_path)) {
    file_path = package_init_path;
    module_type = CN_MODULE_TYPE_PACKAGE;
} else {
    if (package_init_path) {
        free(package_init_path);
    }

    // 2. 再尝试 module_path.cn
    size_t path_len = strlen(module_path);
    file_path = (char *)malloc(path_len + 4);  // ".cn" + '\0'
    if (!file_path) {
        free(module_path);
        return NULL;
    }
    sprintf(file_path, "%s.cn", module_path);
    if (!file_exists(file_path)) {
        free(file_path);
        free(module_path);
        return NULL;
    }

    module_type = CN_MODULE_TYPE_FILE;
}

free(module_path);
```

这样就满足了你在需求 1/2 中对**相对路径导入**的描述：“`导入 ./模块1;` 尝试查找 `./模块1/__包__.cn`”，并保留了对纯文件模块（例如 `导入 ./module_using_simple;`）的兼容性。

#### 2. 正确标记 `CnModuleMetadata.type`

当前 `cn_module_loader_load_relative` 在创建 metadata 时，固定写死为文件模块：

```c
metadata->type = CN_MODULE_TYPE_FILE;
```

为了和 `cn_module_loader_load` 保持一致（后者使用 `cn_is_package_init_path` 判断），你可以利用前面确定好的 `module_type`：

```c
metadata->type = module_type;
```

或如果你更希望统一逻辑，也可以像 `cn_module_loader_load` 那样，在 `file_path` 决定之后统一检查：

```c
if (cn_is_package_init_path(file_path)) {
    metadata->type = CN_MODULE_TYPE_PACKAGE;
} else {
    metadata->type = CN_MODULE_TYPE_FILE;
}
```

**推荐使用前一种方式**（利用我们刚刚区分包与模块的结果），这样避免重复路径解析。

### 二、为 `导入 模块名;` 接入模块加载器（跨文件查找）

目标：  
- 保持现有 **内联模块** 的行为不变：如果当前文件中有 `模块 工具 { ... }`，则 `导入 工具;` 继续只使用这个内联模块。  
- 如果当前文件里没有这个模块符号，且存在模块加载器，则  
  - 使用搜索路径查找 `模块名.cn`（以及必要时的包目录 `模块名/__包__.cn`），  
  - 编译该外部文件模块/包，  
  - 在全局作用域注册一个 `CN_SEM_SYMBOL_MODULE`，其 `as.module_scope` 指向外部模块的作用域，  
  - 后续 **全量导入 / 选择性导入 / 别名导入** 逻辑保持不变。

现有 “传统导入” 分支在 [`scope_builder.c`](file:///c:/Users/ChenChao/Documents/gitcode/CN_Language/src/semantics/resolution/scope_builder.c#L1814-L1962) 中，大致是：

```c
// 传统导入：查找同文件内的模块
CnSemSymbol *module_sym = cn_sem_scope_lookup_shallow(global_scope,
                                                      import->module_name,
                                                      import->module_name_length);

if (!module_sym) {
    // 报错：模块不存在
}

if (module_sym->kind != CN_SEM_SYMBOL_MODULE || !module_sym->as.module_scope) {
    // 报错：符号不是模块
}

// 后续：别名导入 / 全量导入 / 选择性导入，全部基于 module_sym->as.module_scope
```

#### 1. 增加「找不到内联模块时」的跨文件加载逻辑

在 `module_sym` 查询之后、`if (!module_sym)` 之前，插入一段逻辑：

```c
CnSemSymbol *module_sym = cn_sem_scope_lookup_shallow(global_scope,
                                                      import->module_name,
                                                      import->module_name_length);

// 如果当前文件内没有模块定义，且有模块加载器，则尝试从搜索路径加载外部模块
if (!module_sym && loader && source_file && 
    import->module_name && import->module_name_length > 0) {

    // 1. 拷贝模块名为 C 字符串
    char *module_name_str = (char *)malloc(import->module_name_length + 1);
    if (module_name_str) {
        memcpy(module_name_str, import->module_name, import->module_name_length);
        module_name_str[import->module_name_length] = '\0';

        // 2. 构建模块标识符（单段：模块名）
        CnModuleId *module_id = cn_module_id_create(module_name_str);
        free(module_name_str);

        if (module_id) {
            char *resolved_path = NULL;
            // 3. 通过模块加载器解析路径（查找 模块名.cn 或 包目录/__包__.cn）
            if (cn_module_loader_resolve_path(loader, module_id, &resolved_path)) {
                // 4. 编译外部模块，构建其作用域（支持嵌套导入）
                CnSemScope *external_scope = compile_external_module_recursive(
                    resolved_path,
                    diagnostics,
                    global_scope,
                    loader,
                    source_file);
                free(resolved_path);

                if (external_scope) {
                    // 5. 在全局作用域注册模块符号（命名空间）
                    module_sym = cn_sem_scope_insert_symbol(
                        global_scope,
                        import->module_name,
                        import->module_name_length,
                        CN_SEM_SYMBOL_MODULE);
                    if (module_sym) {
                        module_sym->type = cn_type_new_primitive(CN_TYPE_VOID);
                        module_sym->is_public = 1;
                        module_sym->as.module_scope = external_scope;
                    }
                }
            }

            cn_module_id_free(module_id);
        }
    }
}

// 后面保留原来的 if (!module_sym) 错误处理逻辑
if (!module_sym) {
    cn_support_diag_semantic_error_generic(
        diagnostics,
        CN_DIAG_CODE_SEM_UNDEFINED_IDENTIFIER,
        NULL, 0, 0,
        "语义错误：导入的模块不存在");
    continue;
}
```

这样：

- **优先使用内联模块**：如果当前文件里有 `模块 工具 {...}`，就不会走到 loader 分支。  
- 若内联模块不存在，但 `cnc` 的主程序已经创建了 `CnModuleLoader`（见 [`cli/cnc/main.c`](file:///c:/Users/ChenChao/Documents/gitcode/CN_Language/src/cli/cnc/main.c#L589-L597)），则  
  - `cn_search_config_set_project_root` 已在 `cn_sem_build_scopes_with_loader` 里设置为当前源文件的父目录，所以  
  - `导入 数学;` 会按“当前文件目录 -> 其他搜索路径 -> 标准库路径”的顺序查找 `数学.cn` 和 `数学/__包__.cn`。  

后续的：

- 别名导入：`导入 模块名(别名);`  
- 全量导入：`导入 模块名;`  
- 选择性导入：`导入 模块名 { 成员 };`  

都沿用原有逻辑，统一基于 `module_sym->as.module_scope`，所以对内联模块和外部文件模块/包的行为是一致的。

> 注意：这里我们没有额外输出 “无法编译外部模块” 的诊断，而是让编译子模块过程中产生的语义/语法错误自己报出；如果解析失败，最终会走到 “导入的模块不存在” 的错误分支，行为上是可以接受的。

### 三、语义上的对应关系与你需求的对齐情况

根据上述修改，语义可以这样理解：

- **`导入 ./模块1;`**  
  - 解析为 `module_path` 且 `is_relative = 1`。  
  - 语义阶段调用 `cn_module_loader_load_relative`：  
    - 优先查找 `./模块1/__包__.cn`，存在则视为**包**（`metadata->type = CN_MODULE_TYPE_PACKAGE`）；  
    - 否则查找 `./模块1.cn`，存在则视为**文件模块**；  
  - 得到的模块/包作用域挂到一个 `CN_SEM_SYMBOL_MODULE` 符号上（名字为最后一段 `模块1`），在全局作用域中以“命名空间”的方式使用。  

- **`导入 模块1;`**  
  - 解析为 `module_name`，无 `module_path`。  
  - 语义阶段先查找当前文件内联模块；若没有，再：  
    - 构建 `CnModuleId("模块1")`，  
    - 通过 `cn_module_loader_resolve_path` 在**搜索路径**中查找 `模块1.cn` 或 `模块1/__包__.cn`，  
    - 成功后编译该文件模块/包，并在全局作用域注册一个 `CN_SEM_SYMBOL_MODULE`（同名 `模块1`）。  

这样就满足了你提出的：

- **“导入 ./模块1; 表示导入相对于当前文件的包”**：  
  - 有 `./模块1/__包__.cn` 时，确实会优先作为包处理，并且 `metadata->type` 区分为包。  
- **“导入 模块1; 表示导入当前搜索路径下的模块”**：  
  - 通过 `CnModuleLoader` + 搜索路径配置，查找的是 `模块1.cn`（以及可能存在的包目录）。

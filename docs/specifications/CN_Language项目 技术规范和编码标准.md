# CN_Language项目 技术规范和编码标准

## 1. 概述

### 1.1 项目简介
CN_Language 是一个面向中文用户设计的编程语言项目，旨在提供友好、直观的中文编程体验。项目采用分层架构和SOLID设计原则，确保代码的可维护性、可扩展性和高质量。

### 1.2 文档目的
本文档定义了 CN_Language 项目的技术规范和编码标准，为所有开发人员提供统一的开发指导，确保代码质量、架构一致性和团队协作效率。

### 1.3 适用范围
- 所有新开发的代码必须符合本规范
- 现有代码按实施路线图逐步迁移到本规范
- 所有开发人员、贡献者和维护者都必须遵守

### 1.4 版本信息
- **当前版本**: 2.0.0
- **最后更新**: 2026年1月8日
- **架构版本**: 2.0.0
- **维护者**: CN_Language架构委员会

## 2. 架构原则

### 2.1 分层架构设计
CN_Language 采用三层分层架构，确保清晰的职责分离和单向依赖关系。

#### 2.1.1 基础设施层
- **职责**: 提供基础服务，无业务逻辑
- **模块**: memory（内存管理）、containers（基础数据结构）、utils（工具函数）、platform（平台抽象层）
- **依赖规则**: 只能依赖C标准库和操作系统API，不能依赖任何项目其他层

#### 2.1.2 核心层
- **职责**: 实现编译器核心功能
- **模块**: lexer（词法分析器）、parser（语法分析器）、ast（抽象语法树）、semantic（语义分析）、codegen（代码生成器）、runtime（运行时系统）
- **依赖规则**: 可以依赖基础设施层，不能依赖应用层

#### 2.1.3 应用层
- **职责**: 提供用户界面和交互
- **模块**: cli（命令行界面）、repl（REPL交互环境）、debugger（调试器）、ide-plugin（IDE集成插件）
- **依赖规则**: 可以依赖核心层和基础设施层

#### 2.1.4 依赖方向
- 应用层 → 核心层 → 基础设施层（单向依赖）
- 禁止反向依赖和循环依赖

### 2.2 SOLID设计原则

#### 2.2.1 单一职责原则
- 每个.c文件不超过500行
- 每个函数不超过50行
- 每个模块只负责一个功能领域
- 头文件只暴露必要的接口

#### 2.2.2 开闭原则
- 通过抽象接口支持扩展
- 使用策略模式实现可替换算法
- 模块对修改封闭，对扩展开放

#### 2.2.3 里氏替换原则
- 接口实现必须完全遵守契约
- 子类可无缝替换父类
- 通过单元测试验证替换性

#### 2.2.4 接口隔离原则
- 为不同客户端提供专用接口
- 避免"胖接口"
- 通过接口组合提供灵活的功能集

#### 2.2.5 依赖倒置原则
- 高层模块定义接口
- 低层模块实现接口
- 通过依赖注入管理依赖关系

### 2.3 模块化设计

#### 2.3.1 模块边界定义
- **功能内聚性**: 每个模块专注于单一功能领域
- **数据封装**: 模块内部数据结构对外部不可见
- **接口最小化**: 只暴露必要的API，隐藏实现细节
- **独立编译**: 每个模块可独立编译和测试

#### 2.3.2 接口设计规范
- 使用抽象接口模式，模块间通过接口指针通信
- 接口定义使用前向声明，避免包含具体头文件
- 接口应有明确的输入输出契约
- 接口应定义清晰的错误处理机制

**接口示例**:
```c
typedef struct Stru_LexerInterface_t {
    bool (*initialize)(const char* source);
    Stru_Token_t* (*next_token)(void);
    void (*destroy)(void);
} Stru_LexerInterface_t;
```

## 3. 依赖管理

### 3.1 编译时依赖规则

#### 3.1.1 头文件包含顺序
1. 模块内部头文件
2. C标准库头文件
3. 第三方库头文件
4. 项目其他模块头文件

#### 3.1.2 禁止模式
- 循环包含
- 不必要的包含
- 隐式依赖

### 3.2 运行时依赖
- **动态库**: 优先静态链接，避免运行时依赖
- **系统依赖**: 明确声明最小系统要求
- **功能检测**: 支持运行时功能检测

### 3.3 版本管理

#### 3.3.1 语义化版本
- **格式**: 主版本号.次版本号.修订号
- **规则**:
  - 主版本号：不兼容的API修改
  - 次版本号：向下兼容的功能性新增
  - 修订号：向下兼容的问题修正

#### 3.3.2 兼容性策略
- **向后兼容**: 公共API一旦发布，保持向后兼容
- **废弃策略**: 废弃的API需要标记，至少保留两个主版本
- **版本锁定**: 使用版本锁定文件确保构建一致性

### 3.4 工具支持
- **静态分析**: cppcheck, clang-tidy
- **依赖分析**: 自定义依赖分析脚本
- **图形生成**: 生成依赖关系图
- **循环检测**: 自动检测循环依赖

### 4.2 文件命名规范

#### 4.2.1 C源文件
- **前缀**: CN_
- **扩展名**: .c（源文件）和 .h（头文件）
- **示例**:
  - CN_main.c
  - CN_language.c
  - CN_lexer.c
  - CN_parser.c
  - CN_ast.h

#### 4.2.2 CN源文件
- **扩展名**: .cn
- **示例**:
  - hello_world.cn
  - test_basic.cn

#### 4.2.3 可执行文件
- **主解释器**: cn_language.exe（Windows）/ CN_Language（Linux）
- **调试器**: cn_debugger.exe
- **REPL**: cn_repl.exe

### 4.3 代码命名规范

#### 4.3.1 命名前缀
- **结构体**: Stru_
- **函数**: F_
- **枚举**: Eum_
- **全局变量**: g_
- **常量**: （无前缀，全大写）
- **接口后缀**: _Interface_t

#### 4.3.2 命名示例
```c
// 结构体
typedef struct Stru_LanguageCore_t Stru_LanguageCore_t;

// 接口
typedef struct Stru_LexerInterface_t Stru_LexerInterface_t;

// 函数
Stru_LanguageCore_t* F_create_language_core(void);

// 枚举
enum Eum_TokenType {
    Eum_TOKEN_KEYWORD,
    Eum_TOKEN_IDENTIFIER
};

// 常量
#define CN_MAX_TOKEN_LENGTH 256
```

#### 4.3.3 命名风格
- **结构体名**: 大驼峰命名法（PascalCase），以Stru_前缀开头，以_t结尾
- **接口名**: 大驼峰命名法，以Stru_前缀开头，以_Interface_t结尾
- **函数名**: 系统匈牙利命名法（Systems Hungarian），使用描述性中文或英文名称
- **变量名**: 小驼峰命名法，局部变量使用有意义名称
- **常量名**: 全大写，下划线分隔
- **枚举名**: 大驼峰命名法，枚举值全大写，以枚举名开头

## 5. 编码标准

### 5.1 代码格式化

#### 5.1.1 缩进
- 使用4个空格进行缩进
- 禁止使用制表符

#### 5.1.2 大括号风格
- 使用Allman风格（大括号单独成行）
```c
if (condition)
{
    // 代码块
}
```

#### 5.1.3 行长度
- 每行代码不超过120个字符

#### 5.1.4 空格使用
- **运算符**: 运算符两侧添加空格
- **逗号**: 逗号后添加空格
- **控制语句**: 控制语句关键字后添加空格

### 5.2 注释规范

#### 5.2.1 文件头注释
每个源文件开头应包含文件头注释，包含以下信息：
- 文件名
- 功能描述
- 作者
- 创建日期
- 修改历史
- 版权信息

**示例**:
```c
/******************************************************************************
 * 文件名: CN_lexer.c
 * 功能: CN_Language 词法分析器实现
 * 作者: CN_Language开发团队
 * 创建日期: 2025年12月30日
 * 修改历史:
 *   - 2026年1月5日: 添加关键字支持
 *   - 2026年1月8日: 重构为接口模式
 * 版权: MIT许可证
 ******************************************************************************/
```

#### 5.2.2 函数注释
每个函数前应添加注释说明，使用以下标签：
- @brief: 函数简要描述
- @param: 参数说明
- @return: 返回值说明
- @note: 注意事项

**示例**:
```c
/**
 * @brief 创建新的词法分析器实例
 * @param source 源代码字符串
 * @return 成功返回词法分析器指针，失败返回NULL
 * @note 调用者负责释放返回的指针
 */
Stru_LexerInterface_t* F_create_lexer(const char* source);
```

#### 5.2.3 接口注释
每个接口定义应有详细文档，说明：
- 契约（接口的预期行为）
- 前置条件（调用前必须满足的条件）
- 后置条件（调用后保证的结果）

#### 5.2.4 行内注释
- 复杂逻辑处添加行内注释
- 使用中文清晰说明代码意图
- 重要的架构决策应添加注释说明理由

### 5.3 头文件规范

#### 5.3.1 包含保护
每个头文件必须包含防止重复包含的保护：
```c
#ifndef CN_LEXER_H
#define CN_LEXER_H

// 头文件内容

#endif /* CN_LEXER_H */
```

#### 5.3.2 包含顺序
1. 对应的源文件头文件（如有）
2. C标准库头文件
3. 第三方库头文件
4. 项目其他头文件

#### 5.3.3 前向声明
- 优先使用前向声明，减少编译依赖
- 避免不必要的头文件包含

### 5.4 错误处理

#### 5.4.1 错误对象
- 使用错误对象+返回码组合策略
- 错误对象包含错误码、错误信息和上下文

#### 5.4.2 错误传播
- 错误通过返回值传递，而非全局状态
- 每个函数应明确返回成功或失败状态

#### 5.4.3 资源管理
- 谁创建谁销毁，或使用引用计数
- 确保资源在错误情况下也能正确释放

#### 5.4.4 防御性编程
- 对输入进行验证
- 对错误进行处理，避免程序崩溃

### 5.5 内存管理

#### 5.5.1 统一接口
- 使用统一的内存管理接口（cn_malloc/cn_free等）
- 所有内存分配必须通过统一接口

#### 5.5.2 分配策略
- **系统分配器**: 通用内存分配
- **对象池**: 频繁创建销毁的小对象
- **区域分配器**: 临时内存分配
- **调试分配器**: 调试和内存泄漏检测

#### 5.5.3 内存安全
- 安全的内存分配包装，检测内存错误
- 边界检查，防止缓冲区溢出

#### 5.5.4 所有权
- 明确的数据对象所有权
- 避免悬空指针和内存泄漏

## 6. 开发工作流程

### 6.1 实施路线图

#### 6.1.1 阶段规划
- **阶段1**: 基础设施层重构（3周）
- **阶段2**: 核心层接口定义（4周）
- **阶段3**: 模块边界清晰化（5周）
- **阶段4**: 应用层重构和工具完善（3周）
- **缓冲时间**: 3周缓冲时间
- **总时长**: 18周（90个工作日）

#### 6.1.2 参考文档
- docs/architecture/实施路线图.md

### 6.2 日常开发

#### 6.2.1 代码标准检查
- 运行代码格式检查
- 确保符合单一职责原则
- 验证函数和文件大小限制
- 检查架构合规性

#### 6.2.2 调试输出
- 禁止直接使用printf
- 必须使用CN_LOG宏进行日志输出

#### 6.2.3 模块开发步骤
1. 在docs/architecture/编写架构设计文档
2. 定义抽象接口（interface.h）
3. 在src/实现功能（<500行/文件）
4. 在tests/编写单元测试和集成测试
5. 在docs/api/更新API文档
6. 创建模块README.md和设计文档

### 6.3 测试策略

#### 6.3.1 测试金字塔
- **单元测试**: 大量，覆盖所有公开接口，使用Unity测试框架
- **集成测试**: 中等，测试模块间协作
- **端到端测试**: 少量，完整编译流程测试

#### 6.3.2 覆盖率目标
- **语句覆盖率**: ≥85%
- **分支覆盖率**: ≥70%
- **函数覆盖率**: ≥90%
- **行覆盖率**: ≥85%

#### 6.3.3 性能测试
- 建立关键路径的性能基准
- 定期运行性能测试，确保性能不退化

#### 6.3.4 兼容性测试
- 跨平台支持测试（Windows、Linux、macOS）
- 不同编译器版本测试

### 6.4 代码质量标准

#### 6.4.1 函数标准
- 每个函数不超过50行
- 函数功能单一明确
- 有完整的参数验证
- 圈复杂度≤10

#### 6.4.2 文件标准
- 每个.c文件不超过500行
- 头文件只暴露必要接口
- 文件名清晰表达功能

#### 6.4.3 架构标准
- 遵循分层架构依赖规则
- 模块间通过接口通信
- 无循环依赖
- 依赖关系明确声明

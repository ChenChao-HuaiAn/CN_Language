# CN_Language 架构设计原则

## 文档概述

本文档定义了CN_Language项目的架构设计原则和标准，旨在建立清晰、可维护、可扩展的软件架构，避免后期解耦困难。本架构基于分层设计、模块化原则和SOLID设计原则。

## 1. 架构愿景

### 1.1 核心目标
- **可维护性**：代码结构清晰，易于理解和修改
- **可扩展性**：支持新功能添加而不影响现有架构
- **可测试性**：模块间解耦，便于单元测试和集成测试
- **高性能**：在关键路径优化性能，同时保持架构清晰

### 1.2 设计哲学
- **渐进式演进**：架构应支持渐进式改进，而非一次性重写
- **明确性原则**：架构决策应有明确理由和文档记录
- **实用主义**：在理论完美和实际可行性间取得平衡

## 2. 分层架构设计

### 2.1 三层架构模型

```
┌─────────────────────────────────────────┐
│           应用层 (Application Layer)    │
│  - 命令行界面 (CLI)                     │
│  - REPL交互环境                         │
│  - 调试器界面                           │
│  - 集成开发环境插件                     │
├─────────────────────────────────────────┤
│           核心层 (Core Layer)           │
│  - 编译器前端 (词法分析、语法分析)      │
│  - 编译器后端 (代码生成、优化)          │
│  - 运行时系统 (内存管理、执行引擎)      │
│  - 标准库实现                           │
├─────────────────────────────────────────┤
│       基础设施层 (Infrastructure Layer) │
│  - 基础数据结构 (字符串、数组、哈希表)  │
│  - 内存管理 (分配器、垃圾回收)          │
│  - 错误处理 (异常、日志)                │
│  - 工具库 (UTF-8处理、文件IO、网络)     │
│  - 平台抽象层 (操作系统接口)            │
└─────────────────────────────────────────┘
```

### 2.2 层间依赖规则

1. **单向依赖**：上层可以依赖下层，下层不能依赖上层
2. **接口依赖**：层间通过抽象接口通信，而非具体实现
3. **依赖注入**：通过依赖注入管理层间依赖关系

### 2.3 各层详细规范

#### 2.3.1 基础设施层
- **责任**：提供跨平台的基础服务和工具
- **特点**：无业务逻辑，可独立于项目使用
- **示例模块**：
  - `memory/`：内存分配器、垃圾回收
  - `containers/`：动态数组、哈希表、链表
  - `io/`：文件系统、网络、控制台
  - `utils/`：字符串处理、数学函数、时间处理

#### 2.3.2 核心层
- **责任**：实现CN_Language的核心编译和执行功能
- **特点**：包含主要业务逻辑，依赖基础设施层
- **示例模块**：
  - `lexer/`：词法分析器
  - `parser/`：语法分析器
  - `ast/`：抽象语法树
  - `semantic/`：语义分析
  - `codegen/`：代码生成器
  - `runtime/`：运行时环境

#### 2.3.3 应用层
- **责任**：提供用户界面和交互功能
- **特点**：薄层，主要协调核心层功能
- **示例模块**：
  - `cli/`：命令行界面
  - `repl/`：交互式环境
  - `debugger/`：调试器
  - `ide-plugin/`：IDE集成

## 3. 模块化设计原则

### 3.1 模块划分标准

#### 3.1.1 功能内聚性
- **单一职责**：每个模块只负责一个功能领域
- **高内聚**：模块内部元素紧密相关
- **明确边界**：模块间有清晰的接口边界

#### 3.1.2 模块规模控制
- **源代码文件**：每个.c文件不超过500行
- **头文件**：每个.h文件不超过300行
- **函数**：每个函数不超过50行
- **模块目录**：每个模块目录包含完整的功能实现

### 3.2 模块依赖管理

#### 3.2.1 依赖方向
```
基础设施层 → 无依赖
核心层 → 基础设施层
应用层 → 核心层 + 基础设施层
```

#### 3.2.2 禁止的依赖模式
- **循环依赖**：模块A依赖B，同时B依赖A
- **跨层反向依赖**：低层模块依赖高层模块
- **隐式依赖**：通过全局变量或单例产生的隐式依赖

#### 3.2.3 依赖检测工具
```bash
# 使用工具检查模块依赖
make check-dependencies
# 生成依赖关系图
make dependency-graph
```

### 3.3 模块接口设计

#### 3.3.1 接口定义规范
```c
// 良好示例：明确的抽象接口
#ifndef CN_LEXER_INTERFACE_H
#define CN_LEXER_INTERFACE_H

#include "CN_Token.h"

typedef struct Stru_CN_LexerInterface_t
{
    // 初始化函数
    bool (*initialize)(const char* source, size_t length);
    
    // 核心功能
    Stru_CN_Token_t* (*next_token)(void);
    bool (*has_more_tokens)(void);
    
    // 状态查询
    void (*get_position)(size_t* line, size_t* column);
    const char* (*get_source_name)(void);
    
    // 资源管理
    void (*reset)(void);
    void (*destroy)(void);
    
} Stru_CN_LexerInterface_t;

#endif // CN_LEXER_INTERFACE_H
```

#### 3.3.2 接口设计原则
- **最小接口**：只暴露必要的功能
- **稳定接口**：公共接口一旦发布，保持向后兼容
- **明确契约**：接口应有明确的输入输出约定
- **错误处理**：接口应定义清晰的错误处理机制

## 4. SOLID设计原则应用

### 4.1 单一职责原则 (Single Responsibility Principle)

#### 4.1.1 文件级别
- 每个.c文件只实现一个主要功能
- 文件命名反映其单一职责
- 示例：`CN_token_creation.c`、`CN_token_validation.c`

#### 4.1.2 函数级别
- 每个函数只做一件事
- 函数名明确描述其单一职责
- 复杂功能分解为多个小函数

#### 4.1.3 模块级别
- 每个模块只负责一个业务领域
- 模块目录结构反映其单一职责
- 示例：`src/lexer/`只负责词法分析

### 4.2 开闭原则 (Open-Closed Principle)

#### 4.2.1 扩展机制
- **插件架构**：通过接口支持功能扩展
- **策略模式**：算法可替换而不修改核心逻辑
- **模板方法**：定义算法骨架，子类实现具体步骤

#### 4.2.2 实现示例
```c
// 可扩展的代码生成器接口
typedef struct Stru_CN_CodeGeneratorInterface_t
{
    // 固定接口
    bool (*generate)(Stru_CN_AST_t* ast, const char* output_path);
    
    // 可扩展的优化钩子
    void (*add_optimization_pass)(CN_OptimizationPass_t* pass);
    void (*remove_optimization_pass)(const char* pass_name);
    
} Stru_CN_CodeGeneratorInterface_t;
```

### 4.3 里氏替换原则 (Liskov Substitution Principle)

#### 4.3.1 接口契约
- 所有实现必须完全遵守接口契约
- 子类不能加强前置条件或减弱后置条件
- 子类必须保持父类的不变性

#### 4.3.2 测试验证
```c
// 验证里氏替换原则的测试
void test_liskov_substitution(void)
{
    Stru_CN_LexerInterface_t* lexer1 = create_default_lexer();
    Stru_CN_LexerInterface_t* lexer2 = create_optimized_lexer();
    
    // 两个lexer应该可以通过相同接口使用
    test_lexer_interface(lexer1);
    test_lexer_interface(lexer2);  // 不应失败
    
    destroy_lexer(lexer1);
    destroy_lexer(lexer2);
}
```

### 4.4 接口隔离原则 (Interface Segregation Principle)

#### 4.4.1 专用接口
- 为不同客户端提供专用接口
- 避免"胖接口"强迫客户端依赖不需要的方法
- 通过接口组合提供灵活的功能集

#### 4.4.2 接口分解示例
```c
// 不良设计：胖接口
typedef struct Stru_CN_FatInterface_t
{
    void (*method1)(void);
    void (*method2)(void);
    void (*method3)(void);
    void (*method4)(void);
    void (*method5)(void);
} Stru_CN_FatInterface_t;

// 良好设计：专用接口
typedef struct Stru_CN_ReaderInterface_t
{
    void (*read)(void);
    void (*seek)(size_t position);
} Stru_CN_ReaderInterface_t;

typedef struct Stru_CN_WriterInterface_t
{
    void (*write)(const void* data, size_t size);
    void (*flush)(void);
} Stru_CN_WriterInterface_t;
```

### 4.5 依赖倒置原则 (Dependency Inversion Principle)

#### 4.5.1 抽象依赖
- 高层模块不直接依赖低层模块
- 两者都依赖抽象接口
- 抽象接口定义在高层模块或独立接口层

#### 4.5.2 依赖注入
```c
// 依赖注入容器示例
typedef struct Stru_CN_DependencyContainer_t
{
    Stru_CN_LexerInterface_t* lexer;
    Stru_CN_ParserInterface_t* parser;
    Stru_CN_CodeGeneratorInterface_t* codegen;
    Stru_CN_MemoryAllocator_t* allocator;
} Stru_CN_DependencyContainer_t;

// 通过容器注入依赖
Stru_CN_Compiler_t* create_compiler(Stru_CN_DependencyContainer_t* deps)
{
    Stru_CN_Compiler_t* compiler = allocate_memory(sizeof(Stru_CN_Compiler_t));
    compiler->lexer = deps->lexer;
    compiler->parser = deps->parser;
    compiler->codegen = deps->codegen;
    compiler->allocator = deps->allocator;
    return compiler;
}
```

## 5. 数据流和状态管理

### 5.1 编译器数据流

#### 5.1.1 标准数据流
```
源代码
    ↓
[词法分析器] → Token流
    ↓
[语法分析器] → 抽象语法树(AST)
    ↓
[语义分析器] → 注解AST
    ↓
[中间代码生成] → 三地址码(IR)
    ↓
[优化器] → 优化后IR
    ↓
[目标代码生成] → 目标代码
```

#### 5.1.2 数据传递规则
- **不可变数据**：阶段间传递的数据应尽可能不可变
- **明确所有权**：每个数据对象有明确的所有者
- **错误传播**：错误通过返回值传递，而非全局状态
- **资源管理**：谁创建谁销毁，或使用引用计数

### 5.2 状态管理原则

#### 5.2.1 无状态设计
- 尽可能设计无状态的函数和模块
- 状态通过参数传递，而非全局变量
- 线程安全考虑

#### 5.2.2 状态封装
```c
// 良好设计：封装的状态管理
typedef struct Stru_CN_CompilerState_t
{
    Stru_CN_SourceFile_t* current_file;
    Stru_CN_SymbolTable_t* symbol_table;
    Stru_CN_ErrorList_t* errors;
    Stru_CN_WarningList_t* warnings;
} Stru_CN_CompilerState_t;

// 状态通过参数传递
bool compile_file(const char* filename, Stru_CN_CompilerState_t* state)
{
    state->current_file = open_source_file(filename);
    // ... 编译过程
    return state->errors->count == 0;
}
```

## 6. 错误处理和日志

### 6.1 错误处理策略

#### 6.1.1 错误分类
- **语法错误**：源代码不符合语法规则
- **语义错误**：类型不匹配、未定义标识符等
- **运行时错误**：除零、空指针解引用等
- **系统错误**：内存不足、文件不存在等

#### 6.1.2 错误处理模式
```c
// 使用返回码和错误对象
typedef struct Stru_CN_Error_t
{
    CN_ErrorCode_t code;
    char message[CN_MAX_ERROR_MESSAGE];
    size_t line;
    size_t column;
    const char* filename;
} Stru_CN_Error_t;

CN_Result_t parse_expression(const char* input, Stru_CN_AST_t** ast, 
                             Stru_CN_Error_t* error);
```

### 6.2 日志系统

#### 6.2.1 日志级别
```c
typedef enum Eum_CN_LogLevel_t

    Eum_LOG_LEVEL_TRACE = 5     // 详细跟踪
```

#### 6.2.2 日志使用规范

## 7. 性能考虑

### 7.1 性能优化原则

#### 7.1.1 优化优先级
1. **算法优化**：选择合适的数据结构和算法
2. **内存优化**：减少分配、提高缓存命中率
3. **并行优化**：利用多核并行处理
4. **微优化**：最后考虑指令级优化

#### 7.1.2 性能测量
```c
// 性能测量工具
typedef struct Stru_CN_PerfTimer_t
{
    const char* name;
    uint64_t start_time;
    uint64_t total_time;
    uint32_t call_count;
} Stru_CN_PerfTimer_t;

// 使用示例
CN_PERF_START("词法分析");
tokenize_source(source_code);
CN_PERF_END("词法分析");
```

### 7.2 内存管理

#### 7.2.1 内存分配策略
- **对象池**：频繁创建销毁的小对象
- **区域分配器**：编译器中间数据
- **引用计数**：共享数据的内存管理
- **垃圾回收**：运行时内存管理

#### 7.2.2 内存安全

## 8. 测试策略

### 8.1 测试金字塔

```
        ┌─────────────────┐
        │   端到端测试     │ (少量)
        └─────────────────┘
                │
        ┌─────────────────┐
        │   集成测试       │ (中等)
        └─────────────────┘
                │
        ┌─────────────────┐
        │   单元测试       │ (大量)
        └─────────────────┘
```

### 8.2 测试类型

#### 8.2.1 单元测试
- **范围**：单个函数或模块
- **目标**：验证内部逻辑正确性
- **工具**：Unity、Check等C单元测试框架

#### 8.2.2 集成测试
- **范围**：多个模块协作
- **目标**：验证模块接口兼容性
- **示例**：词法分析+语法分析集成测试

#### 8.2.3 端到端测试
- **范围**：完整编译流程
- **目标**：验证用户场景
- **示例**：编译并运行.cn文件

### 8.3 测试覆盖率目标
- **语句覆盖率**：≥80%
- **分支覆盖率**：≥70%
- **函数覆盖率**：≥90%
- **行覆盖率**：≥85%

## 10. 实施路线图

### 10.1 第一阶段：基础设施重构（1-2周）
1. 创建基础设施层

# CN_Language 架构决策记录 (ADR)

## 文档概述

本文档记录CN_Language项目的重要架构决策，包括决策背景、考虑因素、决策内容和影响分析。遵循架构决策记录(ADR)模式，确保架构决策的可追溯性和透明度。

## ADR 模板

每个架构决策记录包含以下部分：
1. **标题**：决策的简短描述
2. **状态**：提议、已采纳、已弃用、已替代
3. **背景**：决策的背景和问题描述
4. **考虑因素**：评估的选项和权衡
5. **决策**：选择的方案和理由
6. **影响**：对系统的影响和后果
7. **相关决策**：与此决策相关的其他决策

## 架构决策列表

### ADR-001: 采用分层架构模型

#### 状态
**已采纳** (2026-01-02)

#### 背景
CN_Language项目初期缺乏清晰的架构分层，导致模块间依赖混乱，难以维护和扩展。需要建立明确的架构边界来支持长期发展。

#### 考虑因素

**选项1：单体架构**
- 优点：简单，初期开发快
- 缺点：随着功能增加，代码复杂度指数增长，难以维护

**选项2：微服务架构**
- 优点：高度解耦，独立部署
- 缺点：过度设计，C语言项目不适合微服务，性能开销大

**选项3：分层架构**
- 优点：平衡解耦和性能，适合编译器项目
- 缺点：需要明确的层间接口定义

#### 决策
采用三层分层架构：
1. **基础设施层**：提供基础服务，无业务逻辑
2. **核心层**：实现编译器核心功能
3. **应用层**：提供用户界面和交互

**理由**：
- 分层架构在解耦和性能间取得良好平衡
- 符合编译器项目的自然分层（前端/后端/运行时）
- 支持渐进式重构，可逐步应用

#### 影响
- **正面**：
  - 模块职责清晰，便于团队协作
  - 支持独立测试各层
  - 便于替换实现（如不同优化器）
- **负面**：
  - 初期需要更多设计工作
  - 层间接口需要精心设计
  - 可能引入轻微性能开销

#### 相关决策
- ADR-002: 模块间通信接口设计
- ADR-003: 依赖管理策略

### ADR-002: 模块间通过抽象接口通信

#### 状态
**已采纳** (2026-01-02)

#### 背景
模块间直接依赖具体实现导致紧耦合，难以替换模块实现或进行单元测试。

#### 考虑因素

**选项1：直接函数调用**
- 优点：简单直接，性能最佳
- 缺点：紧耦合，难以测试和替换

**选项2：动态插件系统**
- 优点：高度灵活，运行时可替换
- 缺点：复杂，性能开销大，C语言实现困难

**选项3：抽象接口+静态绑定**
- 优点：解耦同时保持性能，C语言友好
- 缺点：需要接口定义和实现分离

#### 决策
采用抽象接口模式，模块间通过接口指针通信：
```c
// 接口定义
typedef struct Stru_CN_LexerInterface_t {
    bool (*initialize)(const char* source);
    Stru_CN_Token_t* (*next_token)(void);
    void (*destroy)(void);
} Stru_CN_LexerInterface_t;

// 使用接口
bool parse_source(Stru_CN_LexerInterface_t* lexer) {
    lexer->initialize(source_code);
    // 使用lexer接口
}
```

**理由**：
- 在C语言中实现面向接口编程的可行方案
- 保持良好性能（函数指针调用开销小）
- 支持依赖注入和单元测试

#### 影响
- **正面**：
  - 模块解耦，便于单元测试
  - 支持多种实现（如调试版/优化版词法分析器）
  - 便于模拟测试
- **负面**：
  - 代码稍复杂
  - 需要管理接口版本兼容性

#### 相关决策
- ADR-004: 错误处理策略
- ADR-005: 内存管理接口

### ADR-003: 采用SOLID设计原则

#### 状态
**已采纳** (2026-01-02)

#### 背景
项目需要建立统一的设计原则，确保代码质量一致性和长期可维护性。

#### 考虑因素

**选项1：无明确设计原则**
- 优点：无约束，开发自由
- 缺点：代码质量不一致，难以维护

**选项2：自定义设计原则**
- 优点：针对项目定制
- 缺点：需要自行定义和验证

**选项3：采用成熟设计原则**
- 优点：经过验证，有丰富实践指导
- 缺点：可能需要适应C语言特点

#### 决策
采用SOLID设计原则，并针对C语言特点进行适配：

1. **单一职责原则**：
   - 每个.c文件不超过500行
   - 每个函数不超过50行
   - 每个模块只负责一个功能领域

2. **开闭原则**：
   - 通过接口支持扩展
   - 使用策略模式实现可替换算法

3. **里氏替换原则**：
   - 接口实现必须完全遵守契约
   - 通过单元测试验证替换性

4. **接口隔离原则**：
   - 为不同客户端提供专用接口
   - 避免"胖接口"

5. **依赖倒置原则**：
   - 高层模块定义接口
   - 低层模块实现接口
   - 通过依赖注入管理依赖

**理由**：
- SOLID原则经过多年验证，适用于大型项目
- 针对C语言特点进行适配，保持实用性
- 提供明确的设计指导，减少随意性

#### 影响
- **正面**：
  - 提高代码质量和可维护性
  - 便于团队协作和代码审查
  - 支持长期演进
- **负面**：
  - 需要团队学习和适应
  - 初期开发速度可能稍慢

#### 相关决策
- ADR-006: 代码规范标准
- ADR-007: 测试策略

### ADR-004: 统一错误处理策略

#### 状态
**已采纳** (2026-01-02)

#### 背景
项目中错误处理方式不一致，有的使用返回值，有的使用全局错误变量，有的直接退出程序。

#### 考虑因素

**选项1：返回值错误码**
- 优点：简单，C语言标准做法
- 缺点：错误信息有限，调用者可能忽略

**选项2：异常机制**
- 优点：错误传播自动，不会遗漏
- 缺点：C语言不支持，需要复杂模拟

**选项3：错误对象+返回码**
- 优点：丰富错误信息，强制处理
- 缺点：稍复杂，需要错误对象管理

#### 决策
采用错误对象+返回码组合策略：

```c
// 错误对象定义
typedef struct Stru_CN_Error_t {
    CN_ErrorCode_t code;
    char message[CN_MAX_ERROR_MESSAGE];
    size_t line;
    size_t column;
    const char* filename;
} Stru_CN_Error_t;

// 函数返回类型
typedef enum Eum_CN_Result_t {
    Eum_RESULT_SUCCESS = 0,
    Eum_RESULT_ERROR = 1,
    Eum_RESULT_WARNING = 2
} Eum_CN_Result_t;

// 函数签名
Eum_CN_Result_t parse_file(const char* filename, Stru_CN_AST_t** ast, 
                           Stru_CN_Error_t* error);
```

**理由**：
- 平衡简单性和功能性
- 提供丰富错误信息便于调试
- 强制调用者处理错误（通过错误对象参数）

#### 影响
- **正面**：
  - 统一错误处理方式
  - 提供详细错误信息
  - 便于错误日志和报告
- **负面**：
  - 函数签名稍复杂
  - 需要错误对象生命周期管理

#### 相关决策
- ADR-008: 日志系统设计
- ADR-009: 调试支持

### ADR-005: 内存管理策略

#### 状态
**已采纳** (2026-01-02)

#### 背景
项目中内存管理方式混乱，有的使用malloc/free，有的使用自定义分配器，缺乏统一策略。

#### 考虑因素

**选项1：标准malloc/free**
- 优点：简单，标准
- 缺点：性能可能不佳，难以调试内存问题

**选项2：完全自定义内存管理**
- 优点：完全控制，性能优化空间大
- 缺点：复杂，容易出错

**选项3：包装器+策略模式**
- 优点：统一接口，可替换实现
- 缺点：需要抽象层

#### 决策
采用内存管理包装器+多种分配策略：

```c
// 统一内存管理接口
void* cn_malloc(size_t size, const char* purpose);
void* cn_calloc(size_t count, size_t size, const char* purpose);
void* cn_realloc(void* ptr, size_t new_size, const char* purpose);
void cn_free(void* ptr);

// 分配策略
typedef enum Eum_CN_AllocatorType_t {
    Eum_ALLOCATOR_SYSTEM,      // 系统malloc/free
    Eum_ALLOCATOR_POOL,        // 对象池
    Eum_ALLOCATOR_ARENA,       // 区域分配器
    Eum_ALLOCATOR_DEBUG        // 调试分配器（检测内存错误）
} Eum_CN_AllocatorType_t;

// 可配置分配器
void cn_set_allocator(Eum_CN_AllocatorType_t type);
```

**理由**：
- 提供统一接口便于管理和调试
- 支持多种分配策略适应不同场景
- 调试分配器帮助发现内存问题

#### 影响
- **正面**：
  - 统一内存管理接口
  - 支持性能优化（对象池）
  - 便于内存调试和检测
- **负面**：
  - 需要包装层
  - 轻微性能开销

#### 相关决策
- ADR-010: 性能优化策略
- ADR-011: 调试工具集成

### ADR-006: 构建系统选择

#### 状态
**已采纳** (2026-01-02)

#### 背景
需要选择适合C语言项目的构建系统，支持跨平台编译和模块化构建。

#### 考虑因素

**选项1：纯Makefile**
- 优点：简单，广泛支持
- 缺点：复杂项目难以维护，跨平台问题

**选项2：CMake**
- 优点：跨平台好，功能强大
- 缺点：学习曲线，生成文件复杂

**选项3：自定义构建脚本**
- 优点：完全控制
- 缺点：维护成本高

#### 决策
采用Makefile为主，辅以平台特定脚本：

```
构建系统结构：
- Makefile          # 主构建文件
- scripts/compile_linux.sh    # Linux辅助脚本
- scripts/compile_windows.bat # Windows辅助脚本
- scripts/compile_macos.sh    # macOS辅助脚本
```

Makefile关键特性：
- 支持模块化编译
- 自动依赖生成
- 增量编译
- 多配置（Debug/Release）

**理由**：
- Makefile在C语言项目中成熟稳定
- 配合脚本解决跨平台问题
- 团队熟悉，学习成本低

#### 影响
- **正面**：
  - 构建过程透明可控
  - 支持增量编译提高开发效率
  - 跨平台支持
- **负面**：
  - Makefile语法可能复杂
  - 需要维护多个平台脚本

#### 相关决策
- ADR-012: 开发环境配置
- ADR-013: 持续集成流程

### ADR-007: 测试策略

#### 状态
**已采纳** (2026-01-02)

#### 背景
需要建立全面的测试策略，确保代码质量和功能正确性。

#### 考虑因素

**选项1：手动测试**
- 优点：简单，无额外工具
- 缺点：不可靠，不全面，难以维护

**选项2：仅单元测试**
- 优点：覆盖核心逻辑
- 缺点：缺少集成和端到端测试

**选项3：测试金字塔**
- 优点：全面覆盖，平衡成本和效果
- 缺点：需要多种测试工具和框架

#### 决策
采用测试金字塔策略：

1. **单元测试**（大量）：
   - 使用Unity测试框架
   - 每个公开函数都有测试
   - 覆盖率目标：语句≥80%，分支≥70%

2. **集成测试**（中等）：
   - 测试模块间协作
   - 使用自定义测试框架
   - 重点测试接口兼容性

3. **端到端测试**（少量）：
   - 完整编译流程测试
   - 使用.cn示例文件验证
   - 重点测试用户场景

测试目录结构：
```
tests/
├── unit/           # 单元测试
├── integration/    # 集成测试
├── e2e/           # 端到端测试
└── fixtures/      # 测试数据
```

**理由**：
- 测试金字塔平衡测试成本和效果
- 分层测试便于定位问题
- 自动化测试支持持续集成

#### 影响
- **正面**：
  - 提高代码质量和可靠性
  - 支持重构和演进
  - 自动化回归测试
- **负面**：
  - 需要编写和维护测试代码
  - 增加构建时间

#### 相关决策
- ADR-014: 持续集成配置
- ADR-015: 代码质量检查

项目需要全面的文档支持，包括设计文档、API文档、用户指南等。

- 优点：完整支持，便于协作
- 缺点：需要持续维护

#### 决策

### ADR-009: 版本管理策略

#### 状态
**已采纳** (2026-01-02)

#### 背景
需要建立版本管理策略，支持功能演进和兼容性保证。

#### 考虑因素

**选项1：无版本管理**
- 优点：简单
- 缺点：无法管理兼容性，用户困惑

**选项2：语义化版本**
- 优点：标准，明确兼容性
- 缺点：需要严格遵守规则

**选项3：自定义版本规则**
- 优点：针对项目定制
- 缺点：用户需要学习

#### 决策
采用语义化版本(SemVer)：

格式：`主版本号.次版本号.修订号`

- **主版本号**：不兼容的API修改
- **次版本号**：向下兼容的功能性新增
- **修订号**：向下兼容的问题修正

版本管理规则：
1. 公共API一旦发布，保持向后兼容
2. 废弃的API需要标记，至少保留两个主版本
3. 版本号在Makefile和代码中明确定义

**理由**：
- 语义化版本是行业标准
- 明确传达兼容性信息
- 用户和工具都能理解

#### 影响
- **正面**：
  - 明确的兼容性保证
  - 便于依赖管理
  - 支持自动化工具
- **负面**：
  - 需要严格遵守版本规则
  - 废弃API需要长期支持

#### 相关决策
- ADR-018: API稳定性保证
- ADR-019: 发布流程

### ADR-010: 跨平台支持策略

#### 状态
**已采纳** (2026-01-02)

#### 背景
CN_Language需要支持多个平台（Windows、Linux、macOS），确保一致的用户体验。

#### 考虑因素

**选项1：仅支持Linux**
- 优点：开发简单
- 缺点：用户范围受限

**选项2：条件编译**
- 优点：直接，性能好
- 缺点：代码复杂，难以维护

**选项3：平台抽象层**
- 优点：清晰分离，易于维护
- 缺点：需要抽象层设计

#### 决策
采用平台抽象层(PAL)策略：

```
src/platform/
├── pal.h                 # 平台抽象接口
├── pal_linux.c          # Linux实现
├── pal_windows.c        # Windows实现
├── pal_macos.c          # macOS实现
└── pal_stub.c           # 测试桩实现
```

平台抽象接口包括：
- 文件系统操作
- 进程管理
- 控制台I/O
- 时间函数
- 线程和同步

**理由**：
- 清晰分离平台相关代码
- 便于添加新平台支持
- 支持单元测试（使用桩实现）

#### 影响
-

## 1. 概述

### 1.1 设计目标
本设计文档描述了CN_Language语法分析器的错误恢复机制，旨在实现：
1. 提高编译器的容错能力
2. 提供有意义的错误诊断信息
3. 支持短语级错误恢复，避免因单个错误导致整个编译过程终止
4. 提供智能的错误修复建议

### 1.2 设计原则
- **最小干扰原则**: 错误恢复应尽可能少地干扰正常解析流程
- **上下文感知**: 恢复策略应考虑当前解析上下文
- **渐进式恢复**: 从简单到复杂的恢复策略
- **用户友好**: 提供清晰、可操作的错误信息

## 3. 错误恢复策略

### 3.1 短语级恢复 (Phrase-Level Recovery)
短语级恢复是CN_Language的主要错误恢复策略，它允许解析器在遇到错误时继续解析当前短语。

#### 3.1.1 实现机制
```c
// 错误恢复接口
typedef struct Stru_ErrorRecoveryInterface_t {
    // 尝试错误恢复
    bool (*try_recovery)(Stru_ParserInterface_t* parser, Stru_SyntaxError_t* error);
    
    // 获取恢复策略
    Stru_RecoveryStrategy_t (*get_recovery_strategy)(Stru_SyntaxError_t* error);
    
    // 应用恢复策略
    bool (*apply_recovery_strategy)(Stru_ParserInterface_t* parser, 
                                   Stru_RecoveryStrategy_t strategy);
} Stru_ErrorRecoveryInterface_t;
```

#### 3.1.2 支持的恢复操作
1. **插入缺失符号**: 自动插入缺失的分号、括号、逗号等
2. **删除多余符号**: 删除多余的符号
3. **替换错误符号**: 用预期的符号替换错误的符号
4. **跳过错误短语**: 跳过当前错误短语，继续解析

### 3.2 同步恢复 (Synchronization Recovery)
当短语级恢复失败时，使用同步恢复策略。

#### 3.2.1 同步点定义
- 语句结束符（分号）
- 块结束符（右大括号）
- 函数/方法结束
- 模块边界

#### 3.2.2 同步算法
1. 向前扫描令牌流，寻找同步点
2. 丢弃所有中间令牌
3. 在同步点恢复解析

### 3.3 恐慌模式恢复 (Panic Mode Recovery)
当严重错误发生时，使用恐慌模式恢复。

#### 3.3.1 触发条件
- 堆栈溢出
- 内存耗尽
- 无法恢复的语法错误

#### 3.3.2 恢复步骤
1. 清空解析栈
2. 重置解析状态
3. 报告致命错误
4. 尝试继续解析下一个独立单元

## 4. 错误分类和处理

### 4.1 错误分类
| 错误类型 | 严重性 | 恢复策略 | 示例 |
|---------|--------|----------|------|
| 语法错误 | 低 | 短语级恢复 | 缺失分号、括号不匹配 |
| 语义错误 | 中 | 同步恢复 | 类型不匹配、未定义变量 |
| 上下文错误 | 高 | 恐慌模式恢复 | 堆栈溢出、内存错误 |

### 4.2 错误处理流程
```
检测错误 → 分析错误 → 选择恢复策略 → 应用恢复策略 → 报告错误
```

## 5. 实现细节

### 5.1 核心数据结构

#### 5.1.1 语法错误结构
```c
typedef struct Stru_SyntaxError_t {
    Eum_ErrorType type;           // 错误类型
    Eum_ErrorSeverity severity;   // 错误严重性
    Stru_TokenPosition_t position; // 错误位置
    char* message;                // 错误消息
    char* expected;               // 期望的内容
    char* found;                  // 实际找到的内容
    Stru_RecoveryStrategy_t recovery_strategy; // 恢复策略
} Stru_SyntaxError_t;
```

#### 5.1.2 恢复策略结构
```c
typedef struct Stru_RecoveryStrategy_t {
    Eum_RecoveryType type;        // 恢复类型
    union {
        struct {
            Eum_TokenType token_type; // 要插入的令牌类型
            char* token_value;        // 令牌值
        } insert;
        struct {
            int skip_count;           // 跳过的令牌数量
        } skip;
        struct {
            Eum_TokenType from_type;  // 从什么类型
            Eum_TokenType to_type;    // 替换为什么类型
        } replace;
    } action;
    int confidence;                // 恢复置信度 (0-100)
} Stru_RecoveryStrategy_t;
```

### 5.2 关键算法

#### 5.2.1 短语级恢复算法
```c
bool F_try_phrase_level_recovery(Stru_ParserInterface_t* parser, 
                                 Stru_SyntaxError_t* error) {
    // 1. 分析错误上下文
    Stru_ParseContext_t context = F_analyze_error_context(parser, error);
    
    // 2. 选择恢复策略
    Stru_RecoveryStrategy_t strategy = F_select_recovery_strategy(error, context);
    
    // 3. 应用恢复策略
    bool success = F_apply_recovery_strategy(parser, strategy);
    
    // 4. 记录恢复结果
    if (success) {
        F_record_recovery_success(error, strategy);
    } else {
        F_record_recovery_failure(error, strategy);
    }
    
    return success;
}
```

#### 5.2.2 错误上下文分析
```c
Stru_ParseContext_t F_analyze_error_context(Stru_ParserInterface_t* parser,
                                           Stru_SyntaxError_t* error) {
    Stru_ParseContext_t context;
    
    // 获取当前解析状态
    context.current_state = F_get_parser_state(parser);
    
    // 分析前向令牌
    context.lookahead_tokens = F_get_lookahead_tokens(parser, 3);
    
    // 分析解析栈
    context.parse_stack = F_get_parse_stack(parser);
    
    // 分析当前短语
    context.current_phrase = F_get_current_phrase(parser);
    
    return context;
}
```

## 6. 集成设计

### 6.1 与解析器接口集成
错误恢复机制通过解析器接口集成到主解析流程中：

```c
// 在解析器接口中的集成
bool F_consume_token(Stru_ParserInterface_t* parser, 
                     Eum_TokenType expected_type,
                     Stru_SyntaxError_t** error_out) {
    Stru_Token_t* current_token = F_get_current_token(parser);
    
    if (current_token->type != expected_type) {
        // 创建错误对象
        Stru_SyntaxError_t* error = F_create_syntax_error(
            Eum_ERROR_UNEXPECTED_TOKEN,
            current_token->position,
            F_token_type_to_string(expected_type),
            F_token_type_to_string(current_token->type)
        );
        
        // 尝试错误恢复
        if (F_try_error_recovery(parser, error)) {
            // 恢复成功，继续解析
            return true;
        }
        
        // 恢复失败，返回错误
        if (error_out != NULL) {
            *error_out = error;
        } else {
            F_destroy_syntax_error(error);
        }
        
        return false;
    }
    
    return true;
}
```

## 8. 性能考虑

### 8.1 内存使用
- 错误对象池化，减少内存分配

### 8.2 处理速度
- 快速路径：常见错误的快速恢复
- 延迟分析：复杂错误的异步分析
- 并行处理：多个错误的并行恢复

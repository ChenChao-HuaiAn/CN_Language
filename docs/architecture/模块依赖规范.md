# CN_Language 模块依赖规范

## 文档概述

本文档定义了CN_Language项目中模块间的依赖管理规范，旨在建立清晰、可控的依赖关系，避免循环依赖和紧耦合问题。

## 1. 依赖管理原则

### 1.1 核心原则

1. **明确性**：每个模块的依赖必须明确声明
2. **可控性**：依赖关系可检测、可验证、可管理
3. **最小化**：每个模块只依赖必要的其他模块
4. **单向性**：依赖方向应尽可能单向，避免循环

### 1.2 依赖类型

#### 1.2.1 编译时依赖
- **头文件依赖**：通过`#include`引入
- **库依赖**：链接时需要的库文件
- **工具依赖**：构建过程需要的工具

#### 1.2.2 运行时依赖
- **动态库依赖**：运行时加载的共享库
- **数据文件依赖**：运行时需要的数据文件
- **环境依赖**：特定的运行时环境

## 2. 模块依赖规则

### 2.1 分层依赖规则

#### 2.1.1 三层架构依赖方向
```
应用层 (Application Layer)
    ↓ 依赖
核心层 (Core Layer)
    ↓ 依赖
基础设施层 (Infrastructure Layer)
```

#### 2.1.2 具体规则
1. **基础设施层**：
   - 只能依赖C标准库和操作系统API
   - 不能依赖任何项目其他层
   - 示例：`src/infrastructure/`只能包含基础工具

2. **核心层**：
   - 可以依赖基础设施层
   - 不能依赖应用层
   - 核心层内部模块可以有有限依赖
   - 示例：`src/core/lexer/`可以依赖`src/infrastructure/containers/`

3. **应用层**：
   - 可以依赖核心层和基础设施层
   - 应用层内部应尽量减少依赖
   - 示例：`src/application/cli/`可以依赖`src/core/compiler/`

### 2.2 模块内部依赖规则

#### 2.2.1 头文件包含顺序
```c
// 正确的包含顺序
#include "模块内部头文件.h"      // 1. 同模块头文件

#include <标准库头文件.h>        // 2. C标准库
#include <第三方库头文件.h>      // 3. 第三方库

#include "项目其他模块头文件.h"  // 4. 项目其他模块
```

#### 2.2.2 禁止的包含模式
```c
// 错误示例：循环包含
// file_a.h
#include "file_b.h"

// file_b.h  
#include "file_a.h"  // 错误：循环包含

// 错误示例：不必要的包含
#include "完全不需要的头文件.h"  // 错误：增加编译依赖
```

### 2.3 接口依赖规则

#### 2.3.1 接口与实现分离
```
模块A (使用者)
    ↓ 依赖接口
接口定义 (interface.h)
    ↑ 实现接口
模块B (提供者)
```

#### 2.3.2 接口定义规范
```c
// 良好示例：最小接口依赖
// interface.h - 只包含必要的类型定义
#ifndef CN_LEXER_INTERFACE_H
#define CN_LEXER_INTERFACE_H

// 前向声明，避免包含具体头文件
typedef struct Stru_CN_Token_t Stru_CN_Token_t;

typedef struct Stru_CN_LexerInterface_t
{
    // 只使用前向声明的类型
    Stru_CN_Token_t* (*next_token)(void);
    // ...
} Stru_CN_LexerInterface_t;

#endif // CN_LEXER_INTERFACE_H
```

### 4.2 依赖解析工具

#### 4.2.1 依赖解析脚本
```python
#!/usr/bin/env python3
# scripts/dependency_resolver.py

import yaml
import os
from graphlib import TopologicalSorter

class DependencyResolver:
    def __init__(self, project_root):
        self.project_root = project_root
        self.dependencies = {}
        
    def load_module_deps(self, module_path):
        """加载模块依赖配置"""
        dep_file = os.path.join(module_path, "dependencies.yml")
        if os.path.exists(dep_file):
            with open(dep_file, 'r') as f:
                return yaml.safe_load(f)
        return None
    
    def build_dependency_graph(self):
        """构建依赖图"""
        graph = {}
        
        # 遍历所有模块
        for root, dirs, files in os.walk(self.project_root):
            if "dependencies.yml" in files:
                module_name = os.path.relpath(root, self.project_root)
                deps = self.load_module_deps(root)
                
                if deps and 'dependencies' in deps:
                    graph[module_name] = set()
                    for dep in deps['dependencies'].get('compile', []):
                        if dep['type'] == 'internal':
                            graph[module_name].add(dep['module'])
        
        return graph
    
    def check_cycles(self):
        """检查循环依赖"""
        graph = self.build_dependency_graph()
        
        try:
            sorter = TopologicalSorter(graph)
            # 如果拓扑排序成功，说明无循环依赖
            sorter.prepare()
            return True, []
        except CycleError as e:
            return False, e.args[1]  # 返回循环依赖链
```

## 5. 依赖版本管理

### 5.1 版本兼容性规则

#### 5.1.1 语义化版本兼容性
- **主版本变更**：不兼容的API变更，需要显式升级
- **次版本变更**：向后兼容的功能新增，自动兼容
- **修订版本变更**：向后兼容的问题修复，自动兼容

#### 5.1.2 版本约束语法
```
版本约束示例：
1.2.3           # 精确版本
>=1.2.3         # 大于等于指定版本
^1.2.3          # 兼容版本（>=1.2.3 <2.0.0）
~1.2.3          # 近似版本（>=1.2.3 <1.3.0）
1.2.x           # 通配符版本
```

### 5.2 版本锁定机制

#### 5.2.1 版本锁定文件
```yaml
# cn-lock.yml
lockfile_version: "1.0"
generated: "2026-01-02T12:00:00Z"

modules:
  infrastructure/containers:
    version: "1.2.3"
    checksum: "sha256:abc123..."
    dependencies: []
    
  core/lexer:
    version: "2.1.0"
    checksum: "sha256:def456..."
    dependencies:
      - infrastructure/containers@1.2.3
      - infrastructure/utils@1.0.0
```

#### 5.2.2 版本解析策略
```python
def resolve_versions(requested, locked):
    """
    解析版本冲突
    
    策略优先级：
    1. 锁定文件中的版本（如果存在）
    2. 满足所有约束的最新版本
    3. 报告冲突并提示解决
    """
    resolved = {}
    
    for module, constraints in requested.items():
        if module in locked:
            # 使用锁定版本
            resolved[module] = locked[module]
        else:
            # 解析最新兼容版本
            resolved[module] = find_latest_compatible(module, constraints)
    
    return resolved
```

```bash
# 系统要求检查脚本
#!/bin/bash
# scripts/check_system.sh

REQUIRED_TOOLS=("gcc" "make" "python3")
REQUIRED_LIBS=("libc.so.6" "libpthread.so.0")

check_tool() {
    if ! command -v "$1" &> /dev/null; then
        echo "错误: 需要工具 $1"
        return 1
    fi
    return 0
}

check_lib() {
    if ! ldconfig -p | grep -q "$1"; then
        echo "错误: 需要库 $1"
        return 1
    fi
    return 0
}

# 检查所有依赖
for tool in "${REQUIRED_TOOLS[@]}"; do
    check_tool "$tool" || exit 1
done

for lib in "${REQUIRED_LIBS[@]}"; do
    check_lib "$lib" || exit 1
done

echo "系统依赖检查通过"
```

## 7. 依赖优化策略

### 7.1 编译时依赖优化

#### 7.1.1 前向声明
```c
// 优化前：包含完整头文件
#include "CN_Token.h"
#include "CN_Position.h"

// 优化后：使用前向声明
typedef struct Stru_CN_Token_t Stru_CN_Token_t;
typedef struct Stru_CN_Position_t Stru_CN_Position_t;

// 只在实现文件中包含
// CN_lexer.c

```

#### 7.1.2 接口最小化
```c
// 优化前：暴露内部结构
typedef struct Stru_CN_Lexer_t
{
    // 内部字段全部暴露
    char* source;
    size_t position;
    Stru_CN_Token_t* current_token;
    // ...
} Stru_CN_Lexer_t;

// 优化后：不透明指针
typedef struct Stru_CN_Lexer_t Stru_CN_Lexer_t;

// 只通过接口函数访问
```

### 7.2 运行时依赖优化

#### 7.2.1 延迟加载
```c
// 动态库延迟加载
#ifdef CN_USE_DYNAMIC_LOADING
typedef void* (*lib_function_t)(void);

lib_function_t load_function(const char* libname, const char* funcname)
{
    void* handle = dlopen(libname, RTLD_LAZY);
    if (!handle) return NULL;
    
    return dlsym(handle, funcname);
}
#endif
```

#### 7.2.2 功能检测
```c
// 运行时功能检测
bool has_feature(const char* feature)
{
    static struct {
        const char* name;
        bool (*check)(void);
    } features[] = {
        {"utf8", check_utf8_support},
        {"threads", check_thread_support},
        {"simd", check_simd_support},
        {NULL, NULL}
    };
    
    for (int i = 0; features[i].name; i++) {
        if (strcmp(features[i].name, feature) == 0) {
            return features[i].check();
        }
    }
    
    return false;
}
```

## 8. 依赖问题处理

### 8.1 常见依赖问题

#### 8.1.1 循环依赖
```
问题：模块A依赖B，同时B依赖A
解决方案：
1. 提取公共部分到新模块C
2. 使用接口解耦
3. 重构模块职责
```

#### 8.1.2 版本冲突
```
问题：模块X需要库Y版本1.0，模块Z需要库Y版本2.0
解决方案：
1. 寻找兼容版本
2. 使用命名空间隔离
3. 静态链接不同版本
```

#### 8.1.3 缺失依赖
```
问题：运行时缺少必要的库或工具
解决方案：
1. 提供清晰的错误信息
2. 自动下载或安装脚本
3. 静态链接减少依赖
```

## 9. 最佳实践

### 9.1 依赖管理最佳实践

1. **明确声明**：每个模块明确声明所有依赖
2. **定期审查**：定期审查和优化依赖关系
3. **自动化检测**：使用工具自动化检测依赖问题
4. **文档化**：依赖关系和决策要有文档记录
5. **渐进式改进**：逐步优化，避免大规模重构

### 9.2 依赖优化最佳实践

1. **接口最小化**：只暴露必要的接口
2. **编译时优化**：使用前向声明减少编译依赖
3. **运行时灵活**：支持动态加载和功能检测
4. **版本兼容**：遵循语义化版本，保持向后兼容
5. **错误友好**

## 功能完善总结

### 作用域管理架构修复

**修复时间**: 2026-04-13

**问题描述**：
- `scope_builder` 创建的函数作用域和块作用域是临时的，在函数返回后就被丢弃
- `semantic_passes` 重新创建新的作用域，但无法访问 `scope_builder` 插入的符号
- 这导致变量符号的类型信息丢失

**修复方案**：将作用域保存到 AST 节点

**修改的文件**：

1. **`include/cnlang/frontend/ast.h`**
   - 添加 `CnSemScope` 前向声明
   - 在 `CnAstBlockStmt` 中添加 `owning_scope` 字段
   - 在 `CnAstFunctionDecl` 中添加 `owning_scope` 字段

2. **`src/semantics/resolution/scope_builder.c`**
   - 修改 `cn_sem_build_function_scope`：将函数作用域保存到 `function_decl->owning_scope`
   - 修改 `cn_sem_build_block_scope`：将块作用域保存到 `block->owning_scope`
   - 移除作用域释放代码（由 AST 节点拥有）

3. **`src/semantics/checker/semantic_passes.c`**
   - 修改 `cn_sem_resolve_names`：使用 `fn->owning_scope`
   - 修改 `resolve_block_names`：使用 `block->owning_scope`
   - 修改 `cn_sem_check_types`：使用 AST 节点中的作用域
   - 修改 `check_block_types`：使用 AST 节点中的作用域

4. **`src/ir/gen/irgen.c`**
   - 修改 `cn_ir_gen_function`：使用 `func->owning_scope`

5. **`src/frontend/ast/ast.c`**
   - 添加 `#include "cnlang/frontend/semantics.h"`
   - 修改 `cn_frontend_ast_block_free`：释放 `block->owning_scope`
   - 修改 `cn_frontend_ast_function_free`：释放 `function_decl->owning_scope`

**修复效果**：
- 类型信息不再丢失：`semantic_passes` 和 `irgen` 可以访问 `scope_builder` 创建的符号
- 代码更简洁：移除重复的作用域创建和释放代码
- 架构更清晰：作用域生命周期与 AST 节点绑定
